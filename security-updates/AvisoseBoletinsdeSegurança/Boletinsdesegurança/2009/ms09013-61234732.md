Security Bulletin

Microsoft Security Bulletin MS09-013 - Crítica
==============================================

Vulnerabilidades nos Serviços HTTP do Windows poderiam permitir Execução Remota de Código (960803)
--------------------------------------------------------------------------------------------------

Data de publicação: 14 de abril de 2009 | Updated: 29 de abril de 2009

**actualizada:** 1.1

### Informações Gerais

#### Resumo Executivo

Esta actualização de segurança resolve duas vulnerabilidades divulgadas publicamente e duas vulnerabilidades comunicadas de forma privada nos Serviços HTTP do Microsoft Windows (WinHTTP). A vulnerabilidade mais grave poderia permitir a execução remota de código. Um intruso que conseguisse tirar partido desta vulnerabilidade poderia obter o controlo total de um sistema afectado. Um intruso poderia então instalar programas; ver, alterar ou eliminar dados; ou ainda criar novas contas com todos os privilégios. Os utilizadores cujas contas estão configuradas com menos direitos de utilização no sistema podem correr menos riscos do que os utilizadores que trabalham com direitos de utilização de administrador.

Esta actualização de segurança está classificada como Crítica para todas as edições suportadas do Microsoft Windows 2000, Windows XP, Windows Server 2003, Windows Vista e Windows Server 2008. Para mais informações, consulte a subsecção **Software Afectado e Software Não Afectado**, nesta secção.

A actualização de segurança resolve as vulnerabilidades alterando a forma como os Serviços HTTP do Windows processam erros e validam certificados, bem como assegurando que os Serviços HTTP do Windows utilizam correctamente os mecanismos de protecção de reflexão de credenciais NTLM. Para mais informações sobre as vulnerabilidades, consulte a subsecção de Perguntas Mais Frequentes quanto à vulnerabilidade específica na secção seguinte, **Informações sobre as vulnerabilidades**.

**Recomendação.** A maioria dos clientes tem a actualização automática activada e não necessitará de efectuar qualquer acção, porque esta actualização de segurança será transferida e instalada automaticamente. Os clientes que não tenham a actualização automática activada necessitam de verificar as actualizações e instalar esta actualização manualmente. Para informações sobre opções específicas de configuração da actualização automática, consulte o [Artigo 294871 da Base de Dados de Conhecimento da Microsoft](http://support.microsoft.com/kb/294871).

Para administradores e instalações empresariais ou utilizadores finais que pretendam instalar esta actualização de segurança manualmente, a Microsoft recomenda que os clientes apliquem imediatamente a actualização, utilizando software de gestão de actualizações ou verificando as actualizações através do serviço [Microsoft Update](http://update.microsoft.com/microsoftupdate/v6/default.aspx?ln=pt-pt).

Consulte também a secção **Orientações e Ferramentas de Detecção e Implementação**, mais à frente neste boletim.

**Questões conhecidas.** O [Artigo 960803 da Base de Dados de Conhecimento da Microsoft](http://support.microsoft.com/kb/960803) documenta as questões actualmente conhecidas que os clientes podem verificar quando instalam esta actualização de segurança. O artigo também documenta soluções recomendadas para estas questões.

**Onde estão os pormenores sobre as informações do ficheiro?**  
Os pormenores sobre as informações do ficheiro podem ser encontrados no [Artigo 960803 da Base de Dados de Conhecimento da Microsoft](http://support.microsoft.com/kb/960803).

**Esta vulnerabilidade afecta as versões do Windows Server 2008 Service Pack 2 Beta, Windows Vista Service Pack 2 Beta e Windows 7 Beta?**
Sim. Estas vulnerabilidades foram comunicadas após o lançamento do Windows Server 2008 Service Pack 2 Beta, Windows Vista Service Pack 2 Beta e Windows 7 Beta. Aconselha-se os clientes que utilizem estas plataformas a transferirem e a aplicarem a actualização nos seus sistemas.

As actualizações de segurança estão disponíveis no [Microsoft Update](http://update.microsoft.com/microsoftupdate/v6/default.aspx?ln=pt-pt) e no [Windows Update](http://update.microsoft.com/microsoftupdate/v6/default.aspx?ln=pt-pt).

**Como é que esta actualização de segurança está relacionada com o boletim MS09-014?**  
A Vulnerabilidade de Reflexão de Credencias dos Serviços HTTP do Windows, [CVE-2009-0550](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0550), descrita neste boletim de segurança, também afecta o Internet Explorer e a API de Internet do Windows (WinINet). A Microsoft recomenda que os utilizadores do Internet Explorer instalem também a actualização de segurança fornecida pelo boletim [MS09-014](http://go.microsoft.com/fwlink/?linkid=146659), Actualização de Segurança Cumulativa para o Internet Explorer (963027).

**Quais são as questões conhecidas que os clientes poderão experimentar quando instalam esta actualização de segurança?**  
O [Artigo 960803 da Base de Dados de Conhecimento da Microsoft](http://support.microsoft.com/kb/960803) documenta as questões actualmente conhecidas que os clientes podem verificar quando instalam esta actualização de segurança. O artigo também documenta soluções recomendadas para estas questões.

**Por que razão esta actualização resolve diversas vulnerabilidades de segurança comunicadas?**  
Esta actualização contém suporte para diversas vulnerabilidades, porque as modificações necessárias para resolver essas questões estão incluídas nos ficheiros relacionados. Assim, em vez de instalarem diversas actualizações quase iguais, os clientes podem instalar apenas esta actualização.

**Esta actualização contém outras alterações de segurança em termos de funcionalidade?**  
Sim, para resolver a questão descrita em CVE-2009-0089, a Microsoft introduziu uma nova chave de registo, UseFQDNFallbackForCnCompare, em SOFTWARE\\Microsoft\\Windows\\CurrentVersion\\Internet Settings\\WinHttp, que controla o comportamento do WinHTTP ao validar o nome de domínio totalmente qualificado de um anfitrião mediante uma entrada de nome único de um certificado.

Por predefinição, esta chave de registo não existe, o que assegura que o WinHTTP valida totalmente que o nome único (DN) contido no certificado é idêntico ao DN para o qual a ligação foi iniciada. O comportamento anterior pode ser reactivado definindo esta chave de registo para "Verdadeiro". Neste caso, a validação será menos rigorosa e aceitará o facto de o certificado combinar uma alteração do nome através do reencaminhamento ao nível do DNS. Este é o comportamento menos seguro que é predefinido antes da instalação desta actualização de segurança.

Vulnerabilidade de Fluxo Insuficiente de Números Inteiros nos Serviços HTTP do Windows - CVE-2009-0086  
------------------------------------------------------------------------------------------------------
  
<span></span>
Existe uma vulnerabilidade de execução remota de código na forma como os Serviços HTTP do Windows processam valores específicos que são devolvidos por um servidor Web remoto. Um intruso que conseguisse tirar partido desta vulnerabilidade poderia obter o controlo total de um sistema afectado. Um intruso poderia então instalar programas, ver, alterar ou eliminar dados, ou ainda criar novas contas com os mesmos direitos de utilizador que o serviço ou a aplicação que solicita a WinHTTP API a ligar ao servidor Web do intruso.
  
Para visualizar esta vulnerabilidade como uma entrada comum na lista de Vulnerabilidades e Exposições Comuns, consulte [CVE-2009-0086](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0086).
  
#### Factores Atenuantes para a Vulnerabilidade de Fluxo Insuficiente de Números Inteiros nos Serviços HTTP do Windows - CVE-2009-0086
  
A atenuação refere-se a uma definição, uma configuração comum ou uma prática recomendada geral, que existe num estado predefinido, que poderia reduzir a gravidade da exploração de uma vulnerabilidade. Os seguintes factores atenuantes podem ser úteis para a sua situação:
  
-   Nenhuma das edições suportadas do Internet Explorer é afectada por esta vulnerabilidade porque os Serviços HTTP do Windows não são utilizados pelo software.
  
#### Soluções Alternativas para a Vulnerabilidade de Fluxo Insuficiente de Números Inteiros nos Serviços HTTP do Windows - CVE-2009-0086
  
A Microsoft não identificou quaisquer soluções alternativas para esta vulnerabilidade.
  
#### Perguntas Mais Frequentes sobre a Vulnerabilidade de Fluxo Insuficiente de Números Inteiros nos Serviços HTTP do Windows - CVE-2009-0086
  
**Qual é a abrangência desta vulnerabilidade?**    
Esta é uma vulnerabilidade de execução remota de código. Um intruso que conseguisse tirar partido desta vulnerabilidade poderia obter o controlo total de um sistema afectado. Um intruso poderia então instalar programas, ver, alterar ou eliminar dados, ou ainda criar novas contas com os mesmos direitos de utilizador que o serviço ou a aplicação que solicita a WinHTTP API a ligar ao servidor Web do intruso.
  
Esta vulnerabilidade afecta componentes do Windows e aplicações de terceiros que utilizam os Serviços HTTP do Windows e são configurados para ligar a servidores Web remotos e não fidedignos.
  
**O que provoca a vulnerabilidade?**    
A vulnerabilidade é causada pela falta de validação de um parâmetro devolvido por um servidor Web remoto. Isto poderia levar a execução arbitrária de código.
  
**O que são os Serviços HTTP do Windows?**    
Os Serviços HTTP do Windows (WinHTTP) permitem que os programadores com uma interface de programação da aplicação (API) de cliente HTTP enviem pedidos através do protocolo de HTTP para servidores Web. O WinHTTP pode ser utilizado pelos componentes do Microsoft Windows e software de terceiros. O WinHTTP é, entre outros, utilizado pelo serviço Universal Plug and Play (UPnP).
  
**O que é o Universal Plug and Play?**    
O Universal Plug and Play (UPnP) é uma capacidade que permite que dispositivos numa rede descubram outros dispositivos e determinem a forma de trabalhar com eles. O UPnP é mais facilmente compreendido por comparação à capacidade plug-and-play (PnP) que a maior parte dos utilizadores do Windows já conhecem. O PnP permite que o sistema operativo detecte equipamento novo quando o instala num sistema. Por exemplo, quando instala um rato novo no seu computador, o PnP permite que o Windows o detecte, carregue os controladores necessários e comece a utilizá-lo. O UPnP alarga este conceito a dispositivos numa rede, em vez de no próprio sistema local. O UPnP permite que os computadores detectem outros dispositivos na rede e determinem a forma de os utilizar. Por exemplo, um computador podia utilizar o UPnP para detectar se há impressoras na rede que possa utilizar e saber como as utilizar.
  
**O que é o protocolo SSDP (Simple Service Discovery Protocol)?**    
A detecção de dispositivos é o processo através do qual os computadores com capacidade UPnP tomam conhecimento da disponibilidade de dispositivos que podem utilizar e da forma como solicitar serviços dos mesmos. Quando um computador com UPnP arranca, poderão já existir dispositivos na rede que este pode utilizar. Para determinar se é este o caso, o computador envia um pedido de transmissão (chamada uma directiva M-SEARCH), solicitando que um dispositivo com capacidade UPnP na sub-rede lhe responda directamente e forneça informações sobre a sua utilização.
  
Da mesmo forma, quando um dispositivo que suporta UPnP (por exemplo, uma impressora com capacidade UPnP) arranca, poderão já existir computadores com capacidade UPnP na rede que gostariam de o utilizar. O dispositivo transmite uma mensagem (chamada uma directiva NOTIFY) a todos os computadores na sub-rede, anunciando a sua presença na rede e convidando os computadores a utilizar os seus serviços.
  
**Como poderia um intruso utilizar a vulnerabilidade?**    
Um intruso que explorasse esta vulnerabilidade com sucesso poderia obter os mesmos direitos de utilizador que o serviço ou a aplicação que solicita à WinHTTP API para ligar ao servidor Web do intruso. As aplicações executadas em contas configuradas para ter menos direitos de utilizador no sistema poderiam sofrer menos impacto do que as aplicações executadas em contas com direitos de utilizador administrativos.
  
**Como poderia um intruso explorar a vulnerabilidade?**    
Um intruso teria de forçar ou induzir um programa que utilizasse os Serviços HTTP do Windows a ligar-se ao seu servidor Web malicioso. É importante considerar o facto de o serviço Universal Plug and Play (UPnP) utilizar as bibliotecas WinHTTP. Quando este serviço está activado, os utilizadores mal intencionados na sub-rede local podem responder aos pedidos SSDP e, utilizando o WinHTTP, levar o serviço UPnP a ligar a um anfitrião mal intencionado que poderia explorar esta vulnerabilidade. O serviço UPnP é executado na conta LocalService, que é uma conta com privilégios limitados no sistema.
  
**Quais os sistemas mais susceptíveis face a esta vulnerabilidade?**    
Os sistemas afectados são principalmente estações de trabalho e servidores de terminais. Os servidores podem estar mais sujeitos a risco se os administradores permitirem que os utilizadores iniciem sessão nos servidores e executem programas. Contudo, os procedimentos recomendados desencorajam essa acção.
  
**O que faz a actualização?**    
A actualização remove a vulnerabilidade modificando a forma como os Serviços HTTP do Windows processam o erro que resulta numa condição de possível exploração.

#### Perguntas Mais Frequentes sobre a Vulnerabilidade de Nome Incompatível do Certificado dos Serviços HTTP do Windows - CVE- 2009-0089
  
**Qual é a abrangência desta vulnerabilidade?**    
Trata-se de uma vulnerabilidade de falsificação (spoofing) que existe nos Serviços HTTP do Windows. Quando combinado com outros ataques específicos, tais como falsificação (spoofing) do DNS, isto pode permitir que um intruso falsifique com sucesso o certificado digital de um Web site para qualquer aplicação que utilize os Serviços HTTP do Windows.
  
**O que provoca a vulnerabilidade?**    
Quando uma aplicação solicita os serviços HTTP do Windows para efectuar uma ligação a um servidor Web remoto, o WinHTTP valida apenas o nome de domínio totalmente qualificado do URL mediante o certificado do Web site original ao qual o anfitrião tentou ligar-se. As combinações específicas de falsificação (spoofing) do DNS podem reencaminhar uma ligação para um servidor Web diferente que tenha um certificado válido para aquele anfitrião específico, mas não para a página Web à qual o WinHTTP iniciou a ligação. A combinação de falsificação (spoofing) do DNS e esta vulnerabilidade poderiam levar a que o WinHTTP considerasse incorrectamente o certificado para o servidor Web remoto como aceitável. Como o WinHTTP depende da aplicação para a interacção do utilizador, o utilizador poderá não ser avisado desta discrepância pela aplicação.
  
**O que são os Serviços HTTP do Windows?**    
Os Serviços HTTP do Windows (WinHTTP) permitem que os programadores com uma interface de programação da aplicação (API) de cliente HTTP enviem pedidos através do protocolo de HTTP para outros servidores HTTP.
  
**O que são certificados digitais?**    
O certificado digital é uma credencial comum que é utilizada para autenticar a identidade. Uma organização fidedigna atribui um certificado a um indivíduo ou a uma entidade, o alvo do certificado. A organização fidedigna que emite o certificado é uma Autoridade de Certificação (CA) e é conhecida como o emissor do certificado. Uma CA fidedigna emitirá um certificado apenas depois de verificar a identidade do alvo do certificado.
  
**O que é a falsificação (spoofing) do DNS?**    
O Sistema de Nomes de Domínio (DNS) é um dos conjuntos de protocolos padrão da indústria que incluem o TCP/IP. O DNS é implementado usando dois componentes de software: o servidor DNS e o cliente DNS (ou resolução). Ambos os componentes são executados como aplicações de serviços em plano de fundo. Os recursos de rede são identificados por endereços IP numéricos, mas estes endereços IP são difíceis de recordar por parte dos utilizadores de rede. A base de dados do DNS contém registos, que mapeiam nomes alfanuméricos fáceis de utilizar para recursos de rede, para os endereços de IP usados por esses recursos para efeitos de comunicação. Deste modo, o DNS funciona como um dispositivo mnemónico, tornando os recursos de rede mais fáceis de lembrar para utilizadores de rede. Para mais informações e para ver diagramas lógicos que ilustram a forma como o DNS se ajusta a outras tecnologias Windows, consulte o artigo Technet intitulado [O que é o DNS?](http://technet.microsoft.com/en-us/library/cc787921.aspx)
  
A falsificação (spoofing) do DNS é uma categoria ampla de ataques em que, a dado ponto do sistema do DNS, são introduzidas respostas incorrectas do DNS por parte de terceiros que não são normalmente responsáveis por essas respostas. Tal ataque poderá levar a que um domínio ou nome do anfitrião processe um endereço IP que os proprietários desse nome não pretendiam que processasse. Como tal, um anfitrião pode ser induzido a ligar-se a um outro servidor potencialmente malicioso.
  
**Como poderia um intruso utilizar a vulnerabilidade?**    
Um intruso poderia fazer-se passar por um Web site seguro e oferecer conteúdo malicioso à aplicação utilizando os Serviços HTTP do Windows, que confiariam nele como se este fosse originário de um Web site seguro.
  
**Como poderia um intruso explorar a vulnerabilidade?**    
Um intruso teria de combinar esta vulnerabilidade com a falsificação (spoofing) do DNS ou com um ataque semelhante para direccionar a ligação de saída para um servidor sob o seu controlo. Ele ofereceria então um certificado válido para o seu Web site ligado que seria incorrectamente considerado pelos Serviços HTTP do Windows como válido para o Web site anfitrião de destino a que o utilizador tentou inicialmente ligar.
  
**Quais os sistemas mais susceptíveis face a esta vulnerabilidade?**    
Os sistemas afectados são principalmente estações de trabalho e servidores de terminais. Os servidores podem estar mais sujeitos a risco se os administradores permitirem que os utilizadores iniciem sessão nos servidores e executem programas. Contudo, os procedimentos recomendados desencorajam essa acção.
  
**O que faz a actualização?**    
A actualização elimina a vulnerabilidade alterando a forma como os Serviços HTTP do Windows valida certificados que pertencem a um Web site seguro.

Vulnerabilidade de Reflexão de Credenciais dos Serviços HTTP do Windows - CVE-2009-0550  
---------------------------------------------------------------------------------------
  
<span></span>
Existe uma vulnerabilidade de execução remota de código na forma como os Serviços HTTP do Windows processam credencias NTLM quando um utilizador se liga ao servidor Web de um intruso. Esta vulnerabilidade permite a um intruso reproduzir as credenciais do utilizador, devolvendo-as ao intruso, e executar código no contexto de utilizador com sessão iniciada. Se um utilizador tiver sessão iniciada com privilégios administrativos, um intruso que conseguisse tirar partido desta vulnerabilidade poderia obter controlo total sobre um sistema afectado. Um intruso poderia então instalar programas; ver, alterar ou eliminar dados; ou ainda criar novas contas com todos os privilégios. Os utilizadores cujas contas estão configuradas com menos direitos de utilização no sistema podem correr menos riscos do que os utilizadores que trabalham com direitos de utilização de administrador.
  
Para visualizar esta vulnerabilidade como uma entrada comum na lista de Vulnerabilidades e Exposições Comuns, consulte [CVE-2009-0550](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0550).
  
#### Factores Atenuantes para a Vulnerabilidade de Reflexão de Credenciais nos Serviços HTTP do Windows - CVE-2009-0550
  
A atenuação refere-se a uma definição, uma configuração comum ou uma prática recomendada geral, que existe num estado predefinido, que poderia reduzir a gravidade da exploração de uma vulnerabilidade. Os seguintes factores atenuantes podem ser úteis para a sua situação:
  
-   No cenário de um ataque baseado na Web, um intruso poderia alojar um Web site contendo uma página Web que fosse utilizada para explorar esta vulnerabilidade. Além disso, os Web sites afectados e os Web sites que aceitam ou alojam conteúdos ou anúncios fornecidos por utilizadores poderiam conter conteúdos especialmente concebidos que poderiam explorar esta vulnerabilidade. Contudo, em qualquer dos casos, o intruso não teria forma de forçar os utilizadores a visitar estes Web sites. Em vez disso, um intruso teria de convencer os utilizadores a visitar o Web site, levando-os normalmente a clicar numa hiperligação presente numa mensagem de correio electrónico, ou num programa de mensagens instantâneas, que levasse o utilizador para o Web site do intruso.  
-   Um intruso que explorasse com sucesso esta vulnerabilidade poderia obter os mesmos privilégios que o utilizador local. Os utilizadores cujas contas estão configuradas com menos direitos de utilização no sistema podem correr menos riscos do que os utilizadores que trabalham com direitos de utilização de administrador.  
-   Por predefinição, todas as edições suportadas do Windows Vista e Windows Server 2008 têm o perfil definido para público na firewall do Windows, o que reduz a exposição nestas plataformas a esta vulnerabilidade porque, com o perfil público, a firewall bloqueia o tráfego SMB. Uma vez que o cliente necessitaria de executar um serviço que aceitasse a autenticação NTLM para que o ataque fosse bem-sucedido e as credenciais reflectidas, o vector de ataque mais comum seria o SMB.

**Verificar se a actualização foi aplicada**

-   **Microsoft Baseline Security Analyzer**

    Para verificar se uma actualização de segurança foi aplicada num sistema afectado, poderá utilizar a ferramenta Microsoft Baseline Security Analyzer (MBSA). Consulte a secção "**Orientações e ferramentas de detecção e implementação**" mais acima neste boletim para obter mais informações.

-   **Verificação das Versões dos Ficheiros**

    Dado existirem várias edições do Microsoft Windows, os passos que se seguem podem ser diferentes no seu sistema. Caso efectivamente o sejam, consulte a documentação do produto para executar estes passos.

    1.  Clique em **Iniciar** e, em seguida, em **Procurar**.
    2.  No painel **Resultados da Procura**, clique em **Todos os ficheiros e pastas** no **Assistente de Pesquisa**.
    3.  Na caixa **Totalidade ou parte do nome de ficheiro**, escreva um nome de ficheiro a partir da tabela de informações de ficheiros adequada e, em seguida, clique em **Procurar**.
    4.  Na lista de ficheiros, clique com o botão direito do rato no nome de ficheiro a partir da tabela de informações de ficheiros adequada e, em seguida, clique em **Propriedades**.
        **Nota** Dependendo da edição do sistema operativo, ou dos programas instalados no sistema, alguns dos ficheiros incluídos na tabela de informações de ficheiro poderão não estar instalados.
    5.  No separador **Versão**, determine a versão do ficheiro que está instalada no sistema, comparando-a com a versão que é documentada na tabela de informações de ficheiros adequada.
        **Nota** Outros atributos que não a versão do ficheiro poderão sofrer alterações durante a instalação. Comparar outros atributos de ficheiro com as informações contidas na tabela de informações de ficheiro não é um método suportado para verificar a aplicação da actualização. Além disso, em determinados casos, o nome dos ficheiros poderá ser alterado durante a instalação. Se as informações de ficheiro ou de versão não estiverem presentes, utilize um dos outros métodos disponíveis para verificar a instalação da actualização.

-   **Verificação da Chave de Registo**

    Poderá também verificar os ficheiros instalados por esta actualização de segurança consultando as chaves de registo listadas na **Tabela de referência**, nesta secção.

    Estas chaves de registo poderão não conter uma lista completa dos ficheiros instalados. Além disso, estas chaves de registo poderão não ser criadas correctamente se um administrador ou um OEM integrar ou associar esta actualização de segurança aos ficheiros de origem de instalação do Windows.

**Verificar se a actualização foi aplicada**

-   **Microsoft Baseline Security Analyzer**

    Para verificar se uma actualização de segurança foi aplicada num sistema afectado, poderá utilizar a ferramenta Microsoft Baseline Security Analyzer (MBSA). Consulte a secção "**Orientações e ferramentas de detecção e implementação**" mais acima neste boletim para obter mais informações.

-   **Verificação das Versões dos Ficheiros**

    Dado existirem várias edições do Microsoft Windows, os passos que se seguem podem ser diferentes no seu sistema. Caso efectivamente o sejam, consulte a documentação do produto para executar estes passos.

    1.  Clique em **Iniciar** e, em seguida, em **Procurar**.
    2.  No painel **Resultados da Procura**, clique em **Todos os ficheiros e pastas** no **Assistente de Pesquisa**.
    3.  Na caixa **Totalidade ou parte do nome de ficheiro**, escreva um nome de ficheiro a partir da tabela de informações de ficheiros adequada e, em seguida, clique em **Procurar**.
    4.  Na lista de ficheiros, clique com o botão direito do rato no nome de ficheiro a partir da tabela de informações de ficheiros adequada e, em seguida, clique em **Propriedades**.
        **Nota** Dependendo da edição do sistema operativo, ou dos programas instalados no sistema, alguns dos ficheiros incluídos na tabela de informações de ficheiro poderão não estar instalados.
    5.  No separador **Versão**, determine a versão do ficheiro que está instalada no sistema, comparando-a com a versão que é documentada na tabela de informações de ficheiros adequada.
        **Nota** Outros atributos que não a versão do ficheiro poderão sofrer alterações durante a instalação. Comparar outros atributos de ficheiro com as informações contidas na tabela de informações de ficheiro não é um método suportado para verificar a aplicação da actualização. Além disso, em determinados casos, o nome dos ficheiros poderá ser alterado durante a instalação. Se as informações de ficheiro ou de versão não estiverem presentes, utilize um dos outros métodos disponíveis para verificar a instalação da actualização.

-   **Verificação da Chave de Registo**

    Poderá também verificar os ficheiros instalados por esta actualização de segurança consultando as chaves de registo listadas na **Tabela de referência**, nesta secção.

    Estas chaves de registo poderão não conter uma lista completa dos ficheiros instalados. Além disso, estas chaves de registo poderão não ser criadas correctamente se um administrador ou um OEM integrar ou associar esta actualização de segurança aos ficheiros de origem de instalação do Windows.

Quando instala esta actualização de segurança, o programa de instalação verifica se um ou mais ficheiros que estão a ser actualizados no seu sistema foram alvo de uma actualização anterior através de um hotfix da Microsoft.
