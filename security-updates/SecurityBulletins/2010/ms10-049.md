Security Bulletin

Microsoft Security Bulletin MS10-049 - Crítica
==============================================

Vulnerabilidades no SChannel poderiam permitir Execução Remota de Código (980436)
---------------------------------------------------------------------------------

Data de publicação: 10 de agosto de 2010 | Updated: 1 de setembro de 2010

**actualizada:** 1.1

### Informações Gerais

#### Resumo Executivo

Esta actualização de segurança resolve uma vulnerabilidade divulgada publicamente e uma vulnerabilidade comunicada de forma privada no pacote de segurança Secure Channel (SChannel) do Windows. A vulnerabilidade mais grave poderia permitir a execução remota de código se um utilizador visitasse um Web site especialmente concebido para o efeito que fosse concebido para explorar estas vulnerabilidades através de um browser da Web na Internet. Contudo, em qualquer dos casos, o intruso não teria forma de forçar os utilizadores a visitar estes Web sites. Em vez disso, o intruso teria de persuadir os utilizadores a visitar o Web site, levando-os normalmente a clicar numa hiperligação numa mensagem de correio electrónico, ou num pedido de mensagem instantânea que leva o utilizador para o Web site do intruso.

Esta actualização de segurança está classificada como Crítica para todas as edições suportadas do Windows XP e Windows Server 2003 e classificada como Importante para todas as edições suportadas do Windows Vista, Windows Server 2008, Windows 7 e Windows Server 2008 R2. Para mais informações, consulte a subsecção **Software Afectado e Software Não Afectado**, nesta secção.

A actualização de segurança resolve as vulnerabilidades implementando o RFC 5746 e uma validação adicional de respostas SSL devolvidas por um servidor. Para mais informações sobre as vulnerabilidades, consulte a subsecção de Perguntas Mais Frequentes quanto à vulnerabilidade específica na secção seguinte, **Informações sobre as vulnerabilidades**.

Esta actualização de segurança também resolve uma vulnerabilidade descrita previamente no [Aviso de Segurança da Microsoft 977377](http://technet.microsoft.com/security/advisory/977377).

**Recomendação.** A maioria dos clientes tem a actualização automática activada e não necessitará de efectuar qualquer acção, porque esta actualização de segurança será transferida e instalada automaticamente. Os clientes que não tenham a actualização automática activada necessitam de verificar as actualizações e instalar esta actualização manualmente. Para informações sobre opções específicas de configuração da actualização automática, consulte o [Artigo 294871 da Base de Dados de Conhecimento da Microsoft](http://support.microsoft.com/kb/294871).

Para administradores e instalações empresariais ou utilizadores finais que pretendam instalar esta actualização de segurança manualmente, a Microsoft recomenda que os clientes apliquem imediatamente a actualização, utilizando software de gestão de actualizações ou verificando as actualizações através do serviço [Microsoft Update](http://update.microsoft.com/microsoftupdate/v6/default.aspx?ln=pt-pt).

Consulte também a secção **Orientações e Ferramentas de Detecção e Implementação**, mais à frente neste boletim.

**Questões Conhecidas.** Nenhum

Vulnerabilidade de Renegociação de TLS/SSL - CVE-2009-3555  
----------------------------------------------------------
  
<span></span>
Existe uma vulnerabilidade de falsificação (spoof) no protocolo TLS/SSL, implementada no componente de autenticação SChannel do Microsoft Windows. Um intruso que explorasse com sucesso esta vulnerabilidade teria capacidade para introduzir informação numa ligação protegida TLS/SSL, enviando com sucesso tráfego falsificado para o cliente autenticado. Esta actualização resolve uma vulnerabilidade anteriormente discutida no [Aviso de Segurança da Microsoft 977377](http://technet.microsoft.com/security/advisory/977377).
  
Para visualizar esta vulnerabilidade como uma entrada comum na lista de Vulnerabilidades e Exposições Comuns, consulte [CVE-2009-3555](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-3555).
  
#### Factores Atenuantes para a Vulnerabilidade de Renegociação de TLS/SSL - CVE-2009-3555
  
A atenuação refere-se a uma definição, uma configuração comum ou uma prática recomendada geral, que existe num estado predefinido, que poderia reduzir a gravidade da exploração de uma vulnerabilidade. Os seguintes factores atenuantes podem ser úteis para a sua situação:
  
-   Os Web sites que não alojam conteúdos por SSL, mas que apenas servem conteúdo através de ligações HTTP (texto não encriptado) não são afectados.  
-   O Internet Information Services (IIS) 6 e o IIS 7 não permitem a renegociação iniciada pelo cliente. Isto reduz consideravelmente a superfície de ataque. Os sites SSL em que o utilizador introduz um nome de utilizador e uma palavra-passe têm uma baixa probabilidade de ser vulneráveis. O cenário mais comum que poderia permitir a falsificação (spoof) no IIS 6 e no IIS 7 acontece quando os certificados do lado do cliente são utilizados para realizar autenticações mútuas. Este cenário não é comum.  
-   Os clientes apenas são afectados quando um intruso pode efectuar um ataque man-in-the-middle com sucesso, explorando outra vulnerabilidade, tal como um ataque de sub-rede local ou falsificação (spoofing) do DNS.
  
#### Soluções Alternativas para a Vulnerabilidade de Renegociação de TLS/SSL - CVE-2009-3555
  
Uma solução alternativa refere-se a uma definição ou alteração de configuração que não corrige a vulnerabilidade subjacente, mas que ajudaria a bloquear vectores de ataque conhecidos antes de aplicar a actualização. A Microsoft testou as seguintes soluções alternativas e estados na discussão acerca de casos em que uma solução alternativa reduz a funcionalidade:
  
-   **Activar o SSLAlwaysNegoClientCert no IIS 6 e posteriores**
  
    Os servidores Web que executam o IIS 6 e posteriores e que são afectados por exigirem a autenticação mútua com a solicitação de um certificado do cliente podem ser reforçados activando a definição SSLAlwaysNegoClientCert. Tal fará com que o IIS peça ao cliente um certificado aquando da ligação inicial e não exija uma renegociação iniciada pelo servidor.
  
    **Impacto da solução alternativa:** A definição deste sinalizador exigirá que o cliente se autentique antes de carregar qualquer elemento do Web site protegido por SSL. Tal fará com que o browser peça sempre ao utilizador um certificado do cliente perante uma ligação ao Web site protegido por SSL. Se a autenticação mútua *não* for necessária para os conteúdos protegidos por SSL, os utilizadores a quem seja pedido para seleccionar um certificado podem simplesmente ignorar a caixa de diálogo para ver o conteúdo através da ligação SSL, sem ter de seleccionar um certificado do cliente para efeitos de autenticação.
  
    **Para o IIS 6:**
  
    Execute o seguinte comando a partir da pasta "c:\\inetpub\\adminscripts", numa linha de comandos com direitos de administrador / elevados:
  
    `adsutil.vbs SET w3svc/<N>/SSLAlwaysNegoClientCert true`
  
    Em que &lt;N&gt; representa o número do Web site a ser configurado (ou seja, o "Web site predefinido" é 1, o Web site seguinte é 2, etc.). Assim, por exemplo - para proteger o "Web site predefinido" criado pelo IIS, deve ser usado o seguinte comando:
  
    `adsutil.vbs SET w3svc/1/SSLAlwaysNegoClientCert true`
  
    **Para o IIS 7:**
  
    Guarde o texto que se segue num ficheiro com o nome "Enable\_SSL\_Renegotiate\_Workaround.js".
  
    `var vdirObj=GetObject("IIS://localhost/W3svc/1");// substituir 1 nesta linha pelo número do Web site que pretende configurarWScript.Echo("Value of SSLAlwaysNegoClientCert Before: " + vdirObj.SSLAlwaysNegoClientCert);vdirObj.Put("SSLAlwaysNegoClientCert", true);vdirObj.SetInfo();WScript.Echo("Value of SSLAlwaysNegoClientCert After: " + vdirObj.SSLAlwaysNegoClientCert);`
  
    Execute o seguinte comando a partir de uma linha de comandos com direitos de administrador / elevados:
  
    `cscript.exe enable_ssl_renegotiate_workaround.js`
  
#### Perguntas Mais Frequentes sobre a Vulnerabilidade de Renegociação de TLS/SSL - CVE-2009-3555
  
**Qual é a abrangência desta vulnerabilidade?**    
Esta é uma vulnerabilidade de falsificação (spoof). Um intruso que explorasse com sucesso esta vulnerabilidade teria capacidade para introduzir informação numa ligação protegida TLS/SSL, enviando com sucesso tráfego falsificado para o cliente autenticado.
  
**O que provoca a vulnerabilidade?**    
O protocolo TLS, mencionado em RFC 2246, descreve a funcionalidade de renegociação, que permite que qualquer um dos elementos renegoceie os parâmetros de uma ligação protegida em qualquer altura. Um intruso que possa tirar partido de outro ataque, tal como falsificação (spoofing) do DNS ou ataque de sub-rede local, para se tornar um man-in-the-middle em tal ligação, poderia abusar dessa funcionalidade de renegociação para adicionar comandos específicos da aplicação a uma sessão de TLS válida a ser configurada.
  
Em cenários específicos, o servidor pode executar esses comandos no contexto do cliente com autenticação TLS/SSL. No Windows, o TLS e o SSL são implementados pelo pacote de segurança Secure Channel (SChannel).
  
**Por que motivo não é afectada por esta vulnerabilidade uma instalação predefinida do IIS 6.0 e posteriores**  
Esta vulnerabilidade apenas pode ser explorada quando qualquer um dos elementos da ligação tiver permissão para renegociar parâmetros de TLS/SSL. O IIS impede qualquer renegociação de TLS/SSL iniciada pelo cliente. Além disso, o IIS apenas iniciará uma tentativa de renegociação do lado do servidor quando configurado para utilizar uma autenticação mútua baseada num certificado, o que é uma configuração não predefinida.
  
Tenha em conta que, em implementações onde a ligação real de TLS/SSL não é terminada directamente no servidor do IIS, mas sim num terminador SSL de terceiros ou num proxy inverso, a ligação ainda poderá estar susceptível a esta vulnerabilidade.
  
**O que é um ataque man-in-the-middle?**  
Um ataque man-in-the-middle ocorre quando um intruso redirecciona a comunicação entre dois utilizadores para passar pelo computador do intruso sem o conhecimento dos dois utilizadores que comunicam entre si. Cada utilizador envolvido na comunicação envia e recebe, inadvertidamente, tráfego do intruso, acreditando sempre estar a comunicar apenas com o utilizador pretendido. No caso desta vulnerabilidade, um man-in-the-middle não pode ler, desencriptar nem alterar a comunicação encriptada entre o cliente e o servidor. Um man-in-the-middle apenas pode introduzir um pedido na sessão de TLS que é executado no contexto do cliente.
  
**O que é a autenticação mútua?**  
A autenticação mútua é uma funcionalidade de segurança em que um processo de cliente deve provar a sua identidade perante um servidor e o servidor deve provar sua identidade perante o cliente, antes de qualquer tráfego da aplicação ser enviado através da ligação cliente/servidor.
  
A autenticação mútua exige que o cliente e o servidor provem as respectivas identidades entre si antes de executar quaisquer funções da aplicação. A identidade pode ser provada através de meios criptográficos, tais como infra-estruturas de chaves públicas e certificados digitais.
  
**O que é o TLS?**  
O Protocolo Handshake por Transport Layer Security (TLS) é responsável pela autenticação e troca de chaves necessárias para estabelecer ou restabelecer sessões seguras. Ao estabelecer uma sessão segura, o Protocolo Handshake gere o seguinte:
  
Negociação do conjunto de cifras
  
Autenticação do servidor e, opcionalmente, do cliente
  
Troca de informações das chaves de sessão
  
Para mais informações, consulte o artigo da TechNet sobre o [Funcionamento da TLS/SSL](http://technet.microsoft.com/en-us/library/cc783349(ws.10).aspx).
  
**O que é o SSL**  
O SSL é um antecessor do protocolo Transport Layer Security. Executa as mesmas funções e suporta comunicações de rede seguras utilizando uma combinação de tecnologias de chaves secretas e públicas.
  
Para mais informações, consulte o artigo da TechNet sobre o [Funcionamento da TLS/SSL](http://technet.microsoft.com/en-us/library/cc783349(ws.10).aspx).
  
**O que é a renegociação de TLS/SSL?**  
A renegociação é uma funcionalidade dos protocolos TLS e SSL que permite que qualquer um dos elementos renegoceie parâmetros específicos de ligações criptográficas, tais como chaves de encriptação.
  
**O que é o pacote de segurança SChannel?**    
O pacote de segurança Secure Channel (SChannel) é um SSP (Fornecedor de Suporte de Segurança) que implementa os protocolos de autenticação comuns da Internet Secure Sockets Layer (SSL) e Transport Layer Security (TLS). Estes componentes são utilizados para implementar comunicações seguras que suportem diversas aplicações de Internet e de rede, como a navegação na Web. O SChannel faz parte do pacote de segurança que ajuda a fornecer um serviço de autenticação para comunicações seguras entre cliente e servidor.
  
Para mais informações, consulte o artigo MSDN sobre o [Secure Channel](http://msdn.microsoft.com/en-us/library/aa380123(vs.85).aspx).
  
**Como poderia um intruso utilizar a vulnerabilidade?**    
Em implementações vulneráveis, um intruso poderia utilizar esta vulnerabilidade para adicionar conteúdo a um pedido legítimo e iniciado pelo cliente para um servidor no contexto de uma sessão válida com autenticação TLS/SSL. Esta vulnerabilidade não permite que o intruso leia, desencripte ou altere tráfego encriptado entre cliente e servidor.
  
**Como poderia um intruso explorar a vulnerabilidade?**    
Um intruso poderia explorar esta vulnerabilidade configurando um ataque man-in-the-middle entre um cliente e um servidor, interrompendo subsequentemente a sua ligação, assim que acontecesse uma tentativa de renegociação de SSL. Nesta altura, o intruso poderia introduzir um conjunto de dados limitado no caminho protegido, que o servidor iria então interpretar como tendo origem no seu cliente fidedigno.
  
**Quais os sistemas mais susceptíveis a esta vulnerabilidade?**    
Esta vulnerabilidade afecta principalmente os clientes utilizados em browsers da Web e os servidores Web correspondentes, apesar de algumas versões deste ataque poderem afectar outros protocolos TLS/SSL protegidos.
  
**O que faz a actualização?**    
A actualização resolve esta vulnerabilidade implementando o RFC 5746. Este RFC introduz uma extensão de TLS que criptograficamente une as renegociações de TLS à ligação de TLS original, na qual foram executadas. Isto permite que a autenticação seja executada e, mais tarde, validada, evitando assim o ataque.

Vulnerabilidade de Execução Remota de Código em Pedidos de Certificação SChannel Mal Formados - CVE-2010-2566  
-------------------------------------------------------------------------------------------------------------
  
<span></span>
Existe uma vulnerabilidade de execução remota de código na forma como o SChannel de uma máquina cliente valida uma mensagem de pedido de certificação enviada pelo servidor. Um intruso poderia alojar um Web site especialmente concebido para o efeito, destinado a tirar partido desta vulnerabilidade através de um browser da Web na Internet, persuadindo depois um utilizador a ver o Web site em questão. Contudo, em qualquer dos casos, o intruso não teria forma de forçar os utilizadores a visitar estes Web sites. Em vez disso, o intruso teria de persuadir os utilizadores a visitar o Web site, levando-os normalmente a clicar numa hiperligação numa mensagem de correio electrónico, ou num pedido de mensagem instantânea que leva o utilizador para o Web site do intruso.

#### Perguntas Mais Frequentes sobre a Vulnerabilidade de Execução Remota de Código em Pedidos de Certificação SChannel Mal Formados - CVE-2010-2566
  
**Qual é a abrangência desta vulnerabilidade?**    
Esta é uma vulnerabilidade de execução remota de código. Um intruso que conseguisse tirar partido da vulnerabilidade poderia obter o controlo total de um sistema afectado remotamente. Um intruso poderia então instalar programas; ver, alterar ou eliminar dados; ou ainda criar novas contas com todos os privilégios. As tentativas de explorar esta vulnerabilidade iriam, muito provavelmente, resultar numa condição de negação de serviço em vez de execução remota de código.
  
**O que provoca a vulnerabilidade?**    
Esta vulnerabilidade é provocada por uma validação insuficiente do SChannel das mensagens de pedidos de certificação enviadas pelo servidor.

**Como poderia um intruso utilizar a vulnerabilidade?**    
Um intruso que conseguisse tirar partido desta vulnerabilidade com êxito poderia, teoricamente, controlar totalmente o sistema afectado. No entanto, as tentativas de explorar esta vulnerabilidade iriam, muito provavelmente, resultar em Negação do Serviço. O sistema não poderia ligar a Web sites ou a outros recursos de rede que usam SSL ou TLS até que o sistema fosse reiniciado.
  
**Como poderia um intruso explorar a vulnerabilidade?**

Um intruso não teria como obrigar os utilizadores a visitar estes Web sites. **Quais os sistemas mais susceptíveis a esta vulnerabilidade?**    
Esta vulnerabilidade exige que um utilizador tenha sessão iniciada e que visite um Web site para que ocorra qualquer acção nociva. Por conseguinte, quaisquer sistemas onde o browser da Web na Internet ou aplicações que usam SSL ou TLS sejam usados frequentemente, tal como estações de trabalho ou servidores de terminais, estão mais susceptíveis a esta vulnerabilidade.
  
**O que faz a actualização?**    
Esta actualização implementa validação adicional em respostas de SSL devolvidas por um servidor.

**Verificar Se a Actualização Foi Aplicada**

-   **Microsoft Baseline Security Analyzer**

    Para verificar se uma actualização de segurança foi aplicada num sistema afectado, poderá utilizar a ferramenta Microsoft Baseline Security Analyzer (MBSA). Consulte a secção "**Orientações e ferramentas de detecção e implementação**" mais acima neste boletim para obter mais informações.

-   **Verificação das Versões dos Ficheiros**

    Dado existirem várias edições do Microsoft Windows, os passos que se seguem podem ser diferentes no seu sistema. Caso efectivamente o sejam, consulte a documentação do produto para executar estes passos.

    1.  Clique em **Iniciar** e, em seguida, em **Procurar**.
    2.  No painel **Resultados da Procura**, clique em **Todos os ficheiros e pastas** no **Assistente de Pesquisa**.
    3.  Na caixa **Totalidade ou parte do nome de ficheiro**, escreva um nome de ficheiro a partir da tabela de informações de ficheiros adequada e, em seguida, clique em **Procurar**.
    4.  Na lista de ficheiros, clique com o botão direito do rato no nome de ficheiro a partir da tabela de informações de ficheiros adequada e, em seguida, clique em **Propriedades**.
        **Nota** Dependendo da edição do sistema operativo, ou dos programas instalados no sistema, alguns dos ficheiros incluídos na tabela de informações de ficheiro poderão não estar instalados.
    5.  No separador **Versão**, determine a versão do ficheiro que está instalada no sistema, comparando-a com a versão que é documentada na tabela de informações de ficheiros adequada.
        **Nota** Outros atributos que não a versão do ficheiro poderão sofrer alterações durante a instalação. Comparar outros atributos de ficheiro com as informações contidas na tabela de informações de ficheiro não é um método suportado para verificar a aplicação da actualização. Além disso, em determinados casos, o nome dos ficheiros poderá ser alterado durante a instalação. Se as informações de ficheiro ou de versão não estiverem presentes, utilize um dos outros métodos disponíveis para verificar a instalação da actualização.

-   **Verificação da Chave de Registo**

    Poderá também verificar os ficheiros instalados por esta actualização de segurança consultando as chaves de registo listadas na **Tabela de Referência**, nesta secção.

    Estas chaves de registo poderão não conter uma lista completa dos ficheiros instalados. Além disso, estas chaves de registo poderão não ser criadas correctamente se um administrador ou um OEM integrar ou associar esta actualização de segurança aos ficheiros de origem de instalação do Windows.

**Verificar se a Actualização foi aplicada**

-   **Microsoft Baseline Security Analyzer**

    Para verificar se uma actualização de segurança foi aplicada num sistema afectado, poderá utilizar a ferramenta Microsoft Baseline Security Analyzer (MBSA). Consulte a secção "**Orientações e ferramentas de detecção e implementação**" mais acima neste boletim para obter mais informações.

-   **Verificação das Versões dos Ficheiros**

    Dado existirem várias edições do Microsoft Windows, os passos que se seguem podem ser diferentes no seu sistema. Caso efectivamente o sejam, consulte a documentação do produto para executar estes passos.

    1.  Clique em **Iniciar** e, em seguida, em **Procurar**.
    2.  No painel **Resultados da Procura**, clique em **Todos os ficheiros e pastas** no **Assistente de Pesquisa**.
    3.  Na caixa **Totalidade ou parte do nome de ficheiro**, escreva um nome de ficheiro a partir da tabela de informações de ficheiros adequada e, em seguida, clique em **Procurar**.
    4.  Na lista de ficheiros, clique com o botão direito do rato no nome de ficheiro a partir da tabela de informações de ficheiros adequada e, em seguida, clique em **Propriedades**.
        **Nota** Dependendo da edição do sistema operativo, ou dos programas instalados no sistema, alguns dos ficheiros incluídos na tabela de informações de ficheiro poderão não estar instalados.
    5.  No separador **Versão**, determine a versão do ficheiro que está instalada no sistema, comparando-a com a versão que é documentada na tabela de informações de ficheiros adequada.
        **Nota** Outros atributos que não a versão do ficheiro poderão sofrer alterações durante a instalação. Comparar outros atributos de ficheiro com as informações contidas na tabela de informações de ficheiro não é um método suportado para verificar a aplicação da actualização. Além disso, em determinados casos, o nome dos ficheiros poderá ser alterado durante a instalação. Se as informações de ficheiro ou de versão não estiverem presentes, utilize um dos outros métodos disponíveis para verificar a instalação da actualização.

-   **Verificação da Chave de Registo**

    Poderá também verificar os ficheiros instalados por esta actualização de segurança consultando as chaves de registo listadas na **Tabela de Referência**, nesta secção.

    Estas chaves de registo poderão não conter uma lista completa dos ficheiros instalados. Além disso, estas chaves de registo poderão não ser criadas correctamente se um administrador ou um OEM integrar ou associar esta actualização de segurança aos ficheiros de origem de instalação do Windows.

Esta actualização de segurança suporta os seguintes parâmetros de configuração.
  
| Parâmetro                            | Descrição                                                                                                                                            |  
|--------------------------------------|------------------------------------------------------------------------------------------------------------------------------------------------------|  
| **/?, /h, /help**                    | Apresenta ajuda para os parâmetros suportados.                                                                                                       |  
| **/quiet**                           | Elimina a janela de mensagem de estado ou de erro.                                                                                                   |  
| **/norestart**                       | Quando combinado com **/quiet**, o sistema não será reiniciado depois da instalação, mesmo que seja necessário reiniciar para concluir a instalação. |  
| **/warnrestart:&lt;seconds&gt;**     | Quando combinado com **/quiet**, o programa de instalação avisa o utilizador antes de dar lugar ao reinício.                                         |  
| **/promptrestart**                   | Quando combinado com **/quiet**, o programa de instalação solicitará confirmação antes de dar lugar ao reinício.                                     |  
| **/forcerestart**                    | Quando combinado com **/quiet**, o programa de instalação força o encerramento das aplicações e dá lugar ao reinício.                                |  
| **/log:&lt;file name&gt;**           | Activa o registo no ficheiro especificado.                                                                                                           |  
| **/extract:&lt;destination&gt;**     | Extrai o conteúdo do pacote para a pasta de destino.                                                                                                 |  
| **/uninstall /kb:&lt;KB Number&gt;** | Desinstala a actualização de segurança.                                                                                                              |
