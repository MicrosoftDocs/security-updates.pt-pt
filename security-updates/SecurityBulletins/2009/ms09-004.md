Security Bulletin

Microsoft Security Bulletin MS09-004 - Importante
=================================================

Vulnerabilidade no Microsoft SQL Server poderia permitir Execução Remota de Código (959420)
-------------------------------------------------------------------------------------------

Data de publicação: 10 de fevereiro de 2009 | Updated: 18 de março de 2009

**actualizada:** 1.2

### Informações Gerais

#### Resumo Executivo

Esta actualização de segurança resolve uma vulnerabilidade no Microsoft SQL Server comunicada de forma privada. A vulnerabilidade poderia permitir a execução remota de código se utilizadores não fidedignos acedessem a um sistema afectado ou se ocorresse um ataque de injecção de SQL num sistema afectado. Os sistemas com o SQL Server 7.0 Service Pack 4, SQL Server 2005 Service Pack 3 e SQL Server 2008 não são afectados por esta questão.

Esta actualização de segurança está classificada como Importante para versões suportadas do SQL Server 2000, SQL Server 2005 Service Pack 2, Microsoft SQL Server 2000 Desktop Engine (MSDE 2000), SQL Server 2005 Express Edition, Microsoft SQL Server 2000 Desktop Engine (WMSDE) e Windows Internal Database (WYukon). Para mais informações consulte a subsecção **Software Afectado e Software Não Afectado**, nesta secção.

A actualização de segurança resolve a vulnerabilidade validando os parâmetros de entrada transmitidos a um procedimento armazenado expandido. Para mais informações sobre a vulnerabilidade, consulte a subsecção de Perguntas Mais Frequentes (FAQ) quanto à vulnerabilidade específica na secção seguinte, **Informações sobre as vulnerabilidades**.

Esta actualização de segurança também resolve a vulnerabilidade descrita previamente no [Aviso de Segurança da Microsoft 961040](http://technet.microsoft.com/security/advisory/961040).

**Recomendação.** A Microsoft recomenda que os clientes instalem esta actualização na primeira oportunidade.

**Questões conhecidas.** O [Artigo 959420 da Base de Dados de Conhecimento da Microsoft](http://support.microsoft.com/kb/959420) documenta as questões actualmente conhecidas que os clientes podem verificar quando instalam esta actualização de segurança. O artigo também documenta soluções recomendadas para estas questões.

#### Software Afectado e Software Não Afectado

O software que se segue foi testado para determinar quais as versões ou edições afectadas. As outras versões ou edições são antigas, ultrapassaram o respectivo ciclo de vida de suporte ou não são afectadas. Para determinar o ciclo de vida de suporte da versão ou edição do seu software, visite o [Web site do Ciclo de Vida de Suporte Microsoft](http://support.microsoft.com/lifecycle/).

Perguntas mais frequentes relativas a esta actualização de segurança  
--------------------------------------------------------------------
  
<span></span>
**Por que razão foi revisto este boletim a 5 de Março de 2009?**    
A Microsoft reviu este boletim para anunciar uma alteração de lógica de detecção nos pacotes de actualizações para o Microsoft SQL Server 2000 Desktop Engine (WMSDE) no Windows Server 2003 Service Pack 1, Windows Server 2003 Service Pack 2 (KB960082), Windows Server 2003 x64 Edition e Windows Server 2003 x64 Edition Service Pack 2 (KB960082). A alteração de lógica de detecção corrige uma questão relacionada com a instalação que iria resultar na falha de actualização do Microsoft SQL Server 2000 Desktop Engine (WMSDE) nos sistemas que tenham instalado a Ferramenta de Migração do Active Directory (ADMT) ou o Automated Deployment Services (ADS). Trata-se apenas de uma alteração na implementação, que não afecta os ficheiros contidos na actualização inicial. Os clientes que actualizaram os seus sistemas com sucesso não necessitam de reinstalar esta actualização.
  
**Onde estão os pormenores sobre as informações do ficheiro?**    
Os pormenores sobre as informações do ficheiro podem ser encontrados no [Artigo 959420 da Base de Dados de Conhecimento da Microsoft](http://support.microsoft.com/kb/959420).
  
**São oferecidas tanto actualizações GDR como QFE para a minha versão do SQL. Como posso saber que actualização devo utilizar?**    
Em primeiro lugar, determine o número de versão do seu SQL Server. Para mais informações sobre como determinar o número de versão do seu SQL Server, consulte o [Artigo 321185 da Base de Dados de Conhecimento da Microsoft](http://support.microsoft.com/kb/321185).
  
Em segundo lugar, na tabela abaixo, localize a gama de versões em que se enquadra o número de versão do seu SQL Server. A actualização correspondente é a que necessita de ser instalada.
  
**Nota** Se o número de versão do seu SQL Server não se enquadrar em nenhuma das gamas indicadas na tabela abaixo, significa que a versão do seu SQL Server já não é suportada. Faça a actualização para o Service Pack ou produto SQL Server mais recente para poder aplicar esta e outras actualizações de segurança futuras.

**A actualização será oferecida a clusters de SQL Server?**  
Sim. A actualização também será oferecida às instâncias do SQL Server 2000 e SQL Server 2005 em cluster. As actualizações para clusters de SQL Server requerem interacção com o utilizador.

Se o seu cluster de SQL Server 2000 ou SQL Server 2005 tem um nó passivo, a Microsoft recomenda que verifique e aplique primeiro a actualização no nó activo e, depois, verifique e aplique no nó passivo. Quando todos componentes tiverem sido actualizados no nó passivo, a actualização deixará de ser oferecida.

**As tabelas de referência para o SQL Server 2000 e MSDE 2000** **na secção de Informações da Actualização de Segurança mostram que algumas actualizações poderão exigir um reinício. Como posso saber se a minha actualização requer um reinício?**  
Para o SQL Server 2000 e MSDE 2000, é possível ter serviços dependentes. Os serviços dependentes não são reiniciados após a paragem ou reinício do serviço de que dependem. Por esta razão, deverá ser efectuado um reinício após uma instalação bem sucedida para garantir que os serviços dependentes são iniciados. Para mais informações sobre como determinar se está a executar serviços dependentes, consulte o [Artigo 953741 da Base de Dados de Conhecimento da Microsoft](http://support.microsoft.com/kb/953741).

**As actualizações serão aplicadas a instâncias desactivadas do SQL Server 2000, MSDE 2000 e SQL Server 2005?**  
Não. Apenas as instâncias do SQL Server 2000, MSDE 2000 e SQL Server 2005 que não têm um tipo de arranque **desactivado** serão actualizadas pelo programa de instalação. Para mais informações sobre serviços desactivados, consulte o [Artigo 953740 da Base de Dados de Conhecimento da Microsoft](http://support.microsoft.com/kb/953740).

**Estou a usar uma versão anterior do software discutido neste boletim de segurança. O que devo fazer?**  
O software afectado incluído neste boletim foi testado para determinar quais as versões ou edições afectadas. As outras versões ou edições ultrapassaram o respectivo ciclo de vida de suporte. Para determinar o ciclo de vida de suporte da versão ou edição do seu software, visite o [Web site do Ciclo de Vida de Suporte Microsoft](http://support.microsoft.com/lifecycle/).

Os utilizadores que tenham versões ou edições anteriores do software deviam considerar a migração para versões ou edições suportadas como uma prioridade, de modo a poderem proteger os seus sistemas contra a potencial exposição a vulnerabilidades. Para mais informações sobre o Ciclo de Vida dos Produtos Windows, visite o Web site do [Ciclo de Vida de Suporte Microsoft.](http://support.microsoft.com/lifecycle/) Para mais informações sobre o período de suporte alargado a actualizações de segurança para estas versões ou edições de software, visite o Web site de [Suporte Técnico da Microsoft](http://go.microsoft.com/fwlink/?linkid=33328).

Os clientes que necessitem de suporte personalizado para software anterior devem contactar o seu representante de equipa de contas da Microsoft, o seu Gestor Técnico de Contas ou o parceiro ou representante Microsoft adequado para mais informações sobre opções de suporte personalizado. Os clientes que não disponham de um Contrato Alliance, Premier ou Autorizado podem contactar a subsidiária local da Microsoft. Para obter informações de contacto, visite o [Microsoft Worldwide Information](http://go.microsoft.com/fwlink/?linkid=33329), seleccione o país e, em seguida, clique em **Go** para ver uma lista de números de telefone. Quando telefonar, peça para falar com o Gestor de Vendas de Suporte Premier local. Para mais informações, consulte as [Perguntas Mais Frequentes de suporte sobre o ciclo de vida de produto, relativas aos sistemas operativos Windows](http://go.microsoft.com/fwlink/?linkid=33330).

Vulnerabilidade de Substituição de Memória Limitada sp\_replwritetovarbin no SQL Server - CVE-2008-5416
-------------------------------------------------------------------------------------------------------

<span></span>
Existe uma vulnerabilidade de execução remota de código na forma como o SQL Server verifica parâmetros no procedimento armazenado expandido "sp\_replwritetovarbin". A vulnerabilidade poderia permitir a execução remota de código se utilizadores não fidedignos tivessem acesso a um sistema afectado ou se existisse uma vulnerabilidade de injecção de SQL num sistema afectado. Um intruso que conseguisse tirar partido desta vulnerabilidade poderia obter o controlo total de um sistema afectado. Um intruso poderia então instalar programas, ver, alterar ou eliminar dados, ou ainda criar novas contas.

Para visualizar esta vulnerabilidade como uma entrada comum na lista de Vulnerabilidades e Exposições Comuns, consulte o boletim [CVE-2008-5416](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2008-5416).

#### Factores Atenuantes para a Vulnerabilidade de Substituição de Memória Limitada sp\_replwritetovarbin no SQL Server - CVE-2008-5416

A atenuação refere-se a uma definição, uma configuração comum ou uma prática recomendada geral, que existe num estado predefinido, que poderia reduzir a gravidade da exploração de uma vulnerabilidade. Os seguintes factores atenuantes podem ser úteis para a sua situação:

-   Esta questão não afecta edições suportadas do Microsoft SQL Server 7.0 Service Pack 4, Microsoft SQL Server 2005 Service Pack 3 e Microsoft SQL Server 2008.
-   Esta vulnerabilidade não é exposta de forma anónima. Um intruso teria de estar autenticado para explorar esta vulnerabilidade ou de aproveitar uma vulnerabilidade de injecção de SQL numa aplicação Web que permitisse a autenticação.
-   Por predefinição, o MSDE 2000 e o SQL Server 2005 Express não permitem ligações remotas. Um intruso autenticado teria de iniciar o ataque localmente para explorar a vulnerabilidade.

#### Soluções Alternativas para a Vulnerabilidade de Substituição de Memória Limitada sp\_replwritetovarbin no SQL Server - CVE-2008-5416

Uma solução alternativa refere-se a uma definição ou alteração de configuração que não corrige a vulnerabilidade subjacente, mas que ajudaria a bloquear vectores de ataque conhecidos antes de aplicar a actualização. A Microsoft testou as seguintes soluções alternativas e estados na discussão acerca de casos em que uma solução alternativa reduz a funcionalidade:

-   Negar permissões no procedimento armazenado expandido sp\_replwritetovarbin
    Utilize um dos seguintes procedimentos:

    Para negar o acesso ao procedimento armazenado, faça a ligação ao SQL Server como administrador do sistema utilizando o osql.exe ou o sqlcmd.exe, ou através do SQL Server Management Studio, e execute o seguinte script T-SQL:

    `use masterdeny execute on sp_replwritetovarbin to public`

    Para negar o acesso ao procedimento armazenado utilizando o SQL Server como administrador:

    Para o SQL Server 2000:

    1.  Faça a ligação ao SQL Server utilizando o Enterprise Manager como administrador de sistema
    2.  Na janela SQL Server Enterprise Manager, seleccione o servidor que pretende
    3.  Expanda as bases de dados
    4.  Expanda **Master**
    5.  Clique em **Extended Stored Procedures.** Aparece uma lista de procedimentos armazenados.
    6.  Na lista de procedimentos armazenados, clique com o botão direito do rato em **sp\_replwritetovarbin** e seleccione **Propriedades.**
    7.  Na janela de propriedades, clique em **Permissões**
    8.  Em Utilizadores/Funções de bases de dados/Público, procure **Público**, depois clique na caixa na coluna **EXEC**. A caixa transforma-se num **X** vermelho.
    9.  Clique em **OK** duas vezes.

    Para o SQL Server 2005:

    1.  Faça a ligação ao SQL Server utilizando o SQL Server Management Studio como administrador
    2.  Na janela do Explorador de Objectos, seleccione o servidor que pretende
    3.  Expanda as bases de dados e as bases de dados do sistema
    4.  Expanda **Master**
    5.  Expanda **Programação**
    6.  Clique em **Extended Stored Procedures.** Aparece uma lista de procedimentos armazenados.
    7.  Na lista de procedimentos armazenados, clique com o botão direito do rato em **sp\_replwritetovarbin** e seleccione **Propriedades.**
    8.  Na janela de propriedades, clique em **Permissões**
    9.  Clique na opção para **Negar a execução** ao lado das IDs de utilizador pretendidas e clique em **OK.**

    Para aplicar a solução alternativa a todas as instâncias em execução do SQL Server num computador local, está disponível um script em VB e foi fornecido anteriormente no [Artigo 961040 da Base de Dados de Conhecimento da Microsoft](http://support.microsoft.com/kb/961040).

    **Impacto da solução alternativa:** A desactivação do procedimento armazenado expandido sp\_replwritetovarbin evita as actualizações de tabelas de subscrição feitas por todos os utilizadores. O impacto desta solução alternativa só afecta clientes que utilizam replicação transaccional com subscrições actualizáveis. Os clientes que utilizam replicação transaccional com subscrições apenas de leitura, replicação transaccional bidireccional, ou replicação transaccional peer-to-peer não são afectados. Para mais informações sobre replicação transaccional com subscrições actualizáveis, consulte [MSDN](http://msdn.microsoft.com/en-us/library/ms151718(sql.90).aspx).

    **Como anular esta solução alternativa**

    Faça a ligação ao SQL Server como administrador de sistema utilizando o osql.exe ou o sqlcmd.exe, ou através do SQL Server Management Studio, e execute o seguinte script T-SQL:

    `use mastergrant execute on sp_replwritetovarbin to public`

    Para restaurar o acesso ao procedimento armazenado utilizando o SQL Server como administrador:

    2.  Na janela SQL Server Enterprise Manager, seleccione o servidor que pretende
    3.  Expanda as bases de dados

    5.  Clique em **Extended Stored Procedures.** Aparece uma lista de procedimentos armazenados.

    7.  Na janela de propriedades, clique em **Permissões**
    8.  Em Utilizadores/Funções de bases de dados/Público, procure **Público**, depois clique na caixa na coluna **EXEC**. A caixa transforma-se numa marca de verificação verde.
    9.  Clique em **OK** duas vezes.

    Para o SQL Server 2005:

    1.  Faça a ligação ao SQL Server utilizando o SQL Server Management Studio como administrador
    2.  Na janela do Explorador de Objectos, seleccione o servidor que pretende
    3.  Expanda as bases de dados e as bases de dados do sistema
    4.  Expanda **Master**
    5.  Expanda **Programação**

    7.  Na lista de procedimentos armazenados, clique com o botão direito do rato em **sp\_replwritetovarbin** e seleccione **Propriedades.**
    8.  Na janela de propriedades, clique em **Permissões**
    9.  Clique na opção para **Permitir execução** ao lado das IDs de utilizador pretendidas e clique em **OK.**

#### Perguntas Mais Frequentes sobre a Vulnerabilidade de Substituição de Memória Limitada sp\_replwritetovarbin no SQL Server - CVE-2008-5416

**Qual é a abrangência desta vulnerabilidade?**  
Esta é uma vulnerabilidade de execução remota de código. Um intruso que conseguisse tirar partido desta vulnerabilidade poderia obter o controlo total de um sistema afectado. Um intruso poderia então instalar programas; ver, alterar ou eliminar dados; ou ainda criar novas contas com todos os privilégios.

**O que provoca a vulnerabilidade?**  
Existe uma vulnerabilidade de execução remota de código autenticada no procedimento armazenado expandido do MSSQL ("sp\_replwritetovarbin") devido a uma verificação de parâmetros inválida.

**Para que é utilizado o procedimento armazenado expandido sp\_replwritetovarbin?**  
O procedimento armazenado expandido sp\_replwritetovarbin é utilizado pela replicação transaccional com subscritores actualizáveis e apenas quando a subscrição é criada com @update\_mode = 'failover' ou 'queued tran'.

**Como poderia um intruso utilizar a vulnerabilidade?**  
Um intruso que explorasse esta vulnerabilidade poderia executar código arbitrário. Um intruso poderia então instalar programas; ver, alterar ou eliminar dados; ou ainda criar novas contas com todos os privilégios.

**Como poderia um intruso explorar a vulnerabilidade?**  
Um intruso poderia criar uma consulta que activasse o procedimento armazenado expandido vulnerável com um parâmetro inválido, provocando uma sobrecarga da área dinâmica para dados e permitindo a execução de código.

**Quais os sistemas mais susceptíveis face a esta vulnerabilidade?**  
Os servidores configurados para permitir que utilizadores não fidedignos carreguem e executem consultas estão mais susceptíveis à exploração desta vulnerabilidade.

Os clientes e aplicações que utilizem o MSDE 2000 ou o SQL Server 2005 Express correm o risco de sofrer um ataque remoto se tiverem alterado a instalação predefinida para aceitar ligações remotas, se permitirem o acesso de utilizadores não fidedignos ao MSDE 2000 ou ao SQL Server 2005 Express, ou se uma aplicação que utiliza o MSDE 2000 ou o SQL Server 2005 Express tiver uma vulnerabilidade de injecção de SQL.

Todos os sistemas que executem uma versão afectada do software Microsoft SQL Server em que seja permitido a um utilizador malicioso iniciar sessão correm o risco de exploração desta vulnerabilidade. Além disso, aplicações Web com uma base de dados back-end SQL Server estão numa situação de risco se existir uma vulnerabilidade de injecção de SQL. Para mais informações sobre a injecção de SQL, consulte o [Aviso de Segurança da Microsoft 954462](http://technet.microsoft.com/security/advisory/954462).

**O que faz a actualização?**  
A actualização elimina a vulnerabilidade validando correctamente os parâmetros de entrada transmitidos ao procedimento armazenado expandido.

**Quando este boletim de segurança foi publicado, esta vulnerabilidade já tinha sido divulgada publicamente?**  
Sim. Esta vulnerabilidade foi divulgada publicamente. Foi-lhe atribuído o número de Vulnerabilidade e Exposição Comum [CVE-2008-5416](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2008-5416). Esta vulnerabilidade foi previamente descrita no [Aviso de Segurança da Microsoft 961040](http://technet.microsoft.com/security/advisory/961040).

**Quando este boletim de segurança foi publicado, a Microsoft já tinha recebido relatórios indicando que esta vulnerabilidade estava a ser explorada?**  
Não. Quando este boletim de segurança foi publicado pela primeira vez, a Microsoft tinha conhecimento de exemplos divulgados publicamente de código para prova de conceito, mas não tinha recebido quaisquer informações que indicassem que esta vulnerabilidade tivesse sido utilizada publicamente para atacar clientes.

**Verificar se a Actualização foi aplicada**

-   **Microsoft Baseline Security Analyzer**

    Para verificar se uma actualização de segurança foi aplicada num sistema afectado, poderá utilizar a ferramenta Microsoft Baseline Security Analyzer (MBSA). Consulte a secção "Orientações e ferramentas de detecção e implementação" mais acima neste boletim para obter mais informações.

-   **Verificação das Versões dos Ficheiros**

    Dado existirem várias versões do Microsoft Windows, os passos que se seguem podem ser diferentes no seu computador. Caso efectivamente o sejam, consulte a documentação do produto para executar estes passos.

    1.  Clique em **Iniciar** e, em seguida, em **Procurar**.
    2.  No painel **Resultados da Procura**, clique em **Todos os ficheiros e pastas** no **Assistente de Pesquisa**.
    3.  Na caixa **Totalidade ou parte do nome de ficheiro**, escreva um nome de ficheiro a partir da tabela de informações de ficheiros adequada e, em seguida, clique em **Procurar**.
    4.  Na lista de ficheiros, clique com o botão direito do rato no nome de ficheiro a partir da tabela de informações de ficheiros adequada e, em seguida, clique em **Propriedades**.
        **Nota** Dependendo da versão do sistema operativo, ou dos programas instalados, alguns dos ficheiros incluídos na tabela de informações de ficheiro poderão não estar instalados.
    5.  No separador **Versão**, determine a versão do ficheiro que está instalada no computador, comparando-a com a versão que é documentada na tabela de informações de ficheiros adequada.
        **Nota** Outros atributos que não a versão do ficheiro poderão sofrer alterações durante a instalação. Comparar outros atributos de ficheiro com as informações contidas na tabela de informações de ficheiro não é um método suportado para verificar a aplicação da actualização. Além disso, em determinados casos, o nome dos ficheiros poderá ser alterado durante a instalação. Se as informações de ficheiro ou de versão não estiverem presentes, utilize um dos outros métodos disponíveis para verificar a instalação da actualização.

**Verificar se a Actualização foi aplicada**

-   **Microsoft Baseline Security Analyzer**

    Para verificar se uma actualização de segurança foi aplicada num sistema afectado, poderá utilizar a ferramenta Microsoft Baseline Security Analyzer (MBSA). Consulte a secção "Orientações e ferramentas de detecção e implementação" mais acima neste boletim para obter mais informações.

-   **Verificação das Versões dos Ficheiros**

    Dado existirem várias versões do Microsoft Windows, os passos que se seguem podem ser diferentes no seu computador. Caso efectivamente o sejam, consulte a documentação do produto para executar estes passos.

    1.  Clique em **Iniciar** e, em seguida, em **Procurar**.
    2.  No painel **Resultados da Procura**, clique em **Todos os ficheiros e pastas** no **Assistente de Pesquisa**.
    3.  Na caixa **Totalidade ou parte do nome de ficheiro**, escreva um nome de ficheiro a partir da tabela de informações de ficheiros adequada e, em seguida, clique em **Procurar**.
    4.  Na lista de ficheiros, clique com o botão direito do rato no nome de ficheiro a partir da tabela de informações de ficheiros adequada e, em seguida, clique em **Propriedades**.
        **Nota** Dependendo da versão do sistema operativo, ou dos programas instalados, alguns dos ficheiros incluídos na tabela de informações de ficheiro poderão não estar instalados.
    5.  No separador **Versão**, determine a versão do ficheiro que está instalada no computador, comparando-a com a versão que é documentada na tabela de informações de ficheiros adequada.
        **Nota** Outros atributos que não a versão do ficheiro poderão sofrer alterações durante a instalação. Comparar outros atributos de ficheiro com as informações contidas na tabela de informações de ficheiro não é um método suportado para verificar a aplicação da actualização. Além disso, em determinados casos, o nome dos ficheiros poderá ser alterado durante a instalação. Se as informações de ficheiro ou de versão não estiverem presentes, utilize um dos outros métodos disponíveis para verificar a instalação da actualização.
