Security Bulletin

Microsoft Security Bulletin MS09-036 - Importante
=================================================

Vulnerabilidade no ASP.NET no Microsoft Windows poderia permitir Negação de Serviço (970957)
--------------------------------------------------------------------------------------------

Data de publicação: 11 de agosto de 2009 | Updated: 13 de agosto de 2009

**actualizada:** 1.1

### Informações Gerais

#### Resumo Executivo

Esta actualização de segurança resolve uma vulnerabilidade de Negação de Serviço comunicada de forma privada no componente Microsoft .NET Framework do Microsoft Windows. Esta vulnerabilidade apenas pode ser explorada quando o Internet Information Services (IIS) 7.0 se encontra instalado e o ASP.NET está configurado para utilizar o modo integrado em versões afectadas do Microsoft Windows. Um intruso poderia criar pedidos HTTP anónimos especialmente concebidos para o efeito que poderiam levar a que o servidor Web afectado deixasse de responder até que o conjunto aplicacional associado fosse reiniciado. Os clientes que executam conjuntos aplicacionais IIS 7.0 no modo clássico não são afectados por esta vulnerabilidade.

Esta actualização de segurança está classificada como Importante para todas as versões afectadas do Microsoft Windows. Para mais informações consulte a subsecção **Software Afectado e Software Não Afectado**, nesta secção.

A actualização de segurança resolve a vulnerabilidade modificando a forma como o ASP.NET gere o agendamento de pedidos. Para mais informações sobre a vulnerabilidade, consulte a subsecção de Perguntas Mais Frequentes (FAQ) quanto à vulnerabilidade específica na secção seguinte, **Informações sobre as vulnerabilidades**.

**Recomendação.**  A maioria dos clientes tem a actualização automática activada e não necessitará de efectuar qualquer acção, porque esta actualização de segurança será transferida e instalada automaticamente. Os clientes que não tenham a actualização automática activada necessitam de verificar as actualizações e instalar esta actualização manualmente. Para informações sobre opções específicas de configuração da actualização automática, consulte o [Artigo 294871 da Base de Dados de Conhecimento da Microsoft](http://support.microsoft.com/kb/294871).

Para administradores e instalações empresariais, ou utilizadores finais que pretendam instalar esta actualização de segurança manualmente, a Microsoft recomenda que os clientes apliquem a actualização na primeira oportunidade, utilizando software de gestão de actualizações ou verificando as actualizações através do serviço [Microsoft Update](http://update.microsoft.com/microsoftupdate/v6/default.aspx?ln=pt-pt).

Consulte também a secção **Orientações e Ferramentas de Detecção e Implementação**, mais à frente neste boletim.

**Questões conhecidas.** Nenhum

**Por que motivo o Windows** **Vista** **Service Pack 2 e Windows Server 2008 Service Pack 2 não são afectados?**  
A correcção para esta questão já se encontra incluída nas versões do Microsoft .NET Framework fornecidas com o Windows Vista Service Pack 2 e Windows Server 2008 Service Pack 2.

**Por que motivo o Microsoft .NET Framework 3.5 e o Microsoft .NET Framework 3.5 Service Pack 1 são afectados?**  
O Microsoft .NET Framework 3.5 inclui subcomponentes do Microsoft .NET Framework 2.0 Service Pack 1 e .Microsoft NET Framework 3.0 Service Pack 1. O Microsoft .NET Framework 3.5 Service Pack 1 contém subcomponentes do Microsoft .NET Framework 2.0 Service Pack 2 e Microsoft .NET Framework 3.0 Service Pack 2. Como tal, tanto o Microsoft .NET Framework 3.5 como o Microsoft .NET Framework 3.5 Service Pack 1 contêm subcomponentes vulneráveis.

**Por que razão não me está a ser oferecida esta actualização através das Actualizações Automáticas?**  
Esta actualização de segurança está a ser oferecida apenas a sistemas que têm o IIS 7.0 instalado num sistema operativo listado na tabela de Software Afectado. Os sistemas que estão listados na secção de Software Afectado deste boletim, mas nos quais o IIS 7.0 não está instalado, não receberão esta actualização através das Actualizações Automáticas, uma vez que esta vulnerabilidade apenas pode ser explorada quando o IIS 7.0 está instalado.

Vulnerabilidade de Negação de Serviço Remota Não Autenticada no ASP.NET - CVE-2009-1536  
---------------------------------------------------------------------------------------
  
<span></span>
Existe uma vulnerabilidade de Negação de Serviço na forma como o ASP.NET gere o agendamento de pedidos. Um intruso poderia explorar esta vulnerabilidade criando pedidos HTTP anónimos especialmente concebidos para o efeito que poderiam levar a que o servidor Web afectado deixasse de responder até que o conjunto aplicacional associado fosse reiniciado.
  
Para visualizar esta vulnerabilidade como uma entrada comum na lista de Vulnerabilidades e Exposições Comuns, consulte [CVE-2009-1536](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-1536).
  
#### Factores Atenuantes para a Vulnerabilidade de Negação de Serviço Remota Não Autenticada no ASP.NET - CVE-2009-1536
  
A atenuação refere-se a uma definição, uma configuração comum ou uma prática recomendada geral, que existe num estado predefinido, que poderia reduzir a gravidade da exploração de uma vulnerabilidade. Os seguintes factores atenuantes podem ser úteis para a sua situação:
  
-   O código vulnerável no Microsoft .NET Framework é exposto apenas através do IIS 7.0. Nos sistemas que não executam o IIS 7.0, não existem vectores de ataque conhecidos para esta vulnerabilidade.  
-   Os sistemas que executam o ASP.NET no IIS 7.0 no modo clássico não são afectados por esta vulnerabilidade.
  
#### Soluções Alternativas para a Vulnerabilidade de Negação de Serviço Remota Não Autenticada no ASP.NET - CVE-2009-1536
  
As soluções alternativas referem-se a definições ou alterações de configuração que não corrigem a vulnerabilidade subjacente, mas que ajudariam a bloquear vectores de ataque conhecidos antes de aplicar a actualização. A Microsoft testou as seguintes soluções alternativas e estados na discussão acerca de casos em que uma solução alternativa reduz a funcionalidade:
  
-   **Passe para o conjunto de threads nativas do IIS**
  
    Quando o ASP.NET 2.0 está hospedado no IIS 7.0 no modo integrado, o ASP.NET gere os pedidos utilizando o conjunto de threads de runtime de linguagem comum (CLR). Os sistemas nos quais os pedidos são geridos pelo conjunto de threads nativas do IIS 7.0 em vez do conjunto de threads CLR não são afectados por esta questão. Para mais informações, consulte o artigo MSDN, [.NET: The CLR's Thread Pool](http://msdn.microsoft.com/en-us/magazine/cc164139.aspx).
  
    Utilize um dos dois métodos seguintes para configurar o IIS 7.0 de forma a utilizar o conjunto de threads nativas para gerir processos.
  
    **Método 1: Especificar o valor para maxConcurrentRequestsPerCPU no registo**
  
    Esta definição é global e afecta todos os conjuntos aplicacionais. Para configurar esta definição para conjuntos aplicacionais individuais, consulte o Método 2 nesta secção.
  
    **Aviso** A utilização incorrecta do Editor de Registo pode provocar problemas graves que podem vir a obrigar à reinstalação do sistema operativo. A Microsoft não pode garantir que os problemas resultantes da utilização incorrecta do Editor de Registo possam ser resolvidos. A utilização do Editor de Registo é da sua responsabilidade. Para obter informações sobre como editar o registo, deve consultar o tópico de Ajuda "Alterar Chaves e Valores" do Editor de Registo (Regedit.exe) ou os tópicos de Ajuda "Adicionar e Eliminar Informações do Registo" e "Editar Dados do Registo" do Regedt32.exe.
  
    **Nota** Recomendamos a realização de uma cópia de segurança do registo antes de o editar.
  
    1.  Clique em **Iniciar**, clique em **Executar**, escreva **Regedit** na caixa **Abrir** e clique em **OK**.  
    2.  Localize e clique na subchave de registo seguinte:  
        HKEY\_LOCAL\_MACHINE\\SOFTWARE\\Microsoft\\ASP.NET\\2.0.50727.0  
    3.  No menu **Editar**, escolha **Nova**, e clique em **Chave**.  
    4.  Escreva **maxConcurrentRequestsPerCPU** e prima **Enter**.  
    5.  Clique na chave **maxConcurrentRequestsPerCPU**.  
    6.  No menu **Editar**, escolha **Nova** e clique em **DWORD**.  
    7.  Escreva **maxConcurrentRequestsPerCPU** e prima **Enter**.  
    8.  No menu **Editar**, clique em **Modificar** para modificar a entrada de registo **maxConcurrentRequestsPerCPU**.  
    9.  Na caixa **Dados do valor**, escreva **0** e depois clique em **OK**.  
    10. Saia do Editor de registo.  
    11. Utilize o iisreset.exe para reiniciar o Microsoft Internet Information Services (IIS) no servidor Web anfitrião da aplicação, executando o seguinte comando na linha de comandos:  
        iisreset /restart
  
    **Especificar o valor para maxConcurrentRequestsPerCPU através de um ficheiro .REG**
  
    Editor de registo do Windows versão 5.00  
    \[HKEY\_LOCAL\_MACHINE\\SOFTWARE\\Microsoft\\ASP.NET\\2.0.50727.0\]  
    "MaxConcurrentRequestsPerCPU"=dword:00000000
  
    Pode aplicar este ficheiro .reg a sistemas individuais ao clicar duas vezes sobre ele. Pode ainda aplicá-lo através de domínios usando uma Política de Grupo. Para mais informações sobre as Políticas de Grupos, consulte os seguintes Web sites da Microsoft:  
    [Conjunto de Política de Grupos](http://technet2.microsoft.com/windowsserver/en/library/6d7cb788-b31d-4d17-9f1e-b5ddaa6deecd1033.mspx?mfr=true)  
    [O que é o Editor de Objectos da Política de Grupos?](http://technet2.microsoft.com/windowsserver/en/library/47ba1311-6cca-414f-98c9-2d7f99fca8a31033.mspx?mfr=true)  
    [Ferramentas e definições da Política de Grupos Central](http://technet2.microsoft.com/windowsserver/en/library/e926577a-5619-4912-b5d9-e73d4bdc94911033.mspx?mfr=true)
  
    **Impacto da solução alternativa.** Dependendo da natureza das aplicações executadas neste conjunto aplicacional, o desempenho das aplicações poderá ser afectado.
  
    **Como anular esta solução alternativa.**
  
    Anular a solução alternativa manualmente
  
    1.  Clique em **Iniciar**, clique em **Executar**, escreva **Regedit** na caixa **Abrir** e clique em **OK**. Localize e clique na subchave de registo seguinte:  
        HKEY\_LOCAL\_MACHINE\\SOFTWARE\\Microsoft\\ASP.NET\\2.0.50727.0\\maxConcurrentRequestsPerCPU  
    2.  Clique com o botão direito na subchave **maxConcurrentRequestsPerCPU**, clique em **Eliminar** e, em seguida, clique em **Sim**.  
    3.  Saia do Editor de registo. Utilize o iisreset.exe para reiniciar o Microsoft Internet Information Services (IIS) no servidor Web anfitrião da aplicação, executando o seguinte comando a partir da linha de comandos:  
        iisreset /restart

    **Método 2: Especificar o valor para maxConcurrentRequestsPerCPU para um conjunto aplicacional na secção de configuração do ASP.NET de um ficheiro ASPNET.Config**
  
    **Nota** Esta solução alternativa apenas se aplica a sistemas a executar o Microsoft .NET Framework 2.0 SP2 ou Microsoft .NET Framework 3.5 SP1. As versões anteriores do Microsoft .NET Framework não lêem o aspnet.config para esta informação. Este valor substitui o valor especificado para maxConcurrentRequestsPerCPU no registo.
  
    **Nota** Para anular esta solução alternativa, terá de restaurar a versão original do ficheiro aspnet.config. Antes de editar este ficheiro, crie uma cópia de segurança.
  
    Abra o ficheiro aspnet.config para o conjunto aplicacional e introduza valores novos para o parâmetro maxConcurrentRequestsPerCPU, tal como se segue:
  
    &lt;system.web&gt;  
    &lt;applicationPool maxConcurrentRequestsPerCPU="0"/&gt;  
    &lt;/system.web&gt;
  
    **Impacto da solução alternativa.** Dependendo da natureza das aplicações executadas neste conjunto aplicacional, o desempenho das aplicações poderá ser afectado.
  
    **Como anular esta solução alternativa.** Restaure a versão original do ficheiro aspnet.config.
  
#### Perguntas Mais Frequentes sobre a Vulnerabilidade de Negação de Serviço Remota Não Autenticada no ASP.NET - CVE-2009-1536
  
**Qual é a abrangência desta vulnerabilidade?**    
Esta é uma vulnerabilidade de Negação de Serviço que poderia permitir que um intruso fizesse com que um servidor Web deixasse de responder.
  
**O que provoca a vulnerabilidade?**    
Esta vulnerabilidade é provocada pela gestão incorrecta de agendamento de pedidos por parte do ASP.NET.
  
**Como poderia um intruso utilizar a vulnerabilidade?**    
Um intruso que explorasse esta vulnerabilidade com sucesso poderia fazer com que o conjunto aplicacional no servidor Web afectado deixasse de responder até que esse conjunto fosse reiniciado. Por esse motivo, as páginas Web que utilizam ASP.NET no mesmo conjunto aplicacional deixariam de estar disponíveis e devolveriam um erro HTTP.
  
Nenhuma funcionalidade não-ASP.NET do servidor é afectada quando um servidor vulnerável a esta questão é atacado. Por exemplo, as páginas HTML continuariam a carregar normalmente.
  
Esta é uma vulnerabilidade de negação de serviço a nível de aplicação. Para recuperar de um ataque bem sucedido, o IIS tem de ser reiniciado ou o conjunto aplicacional afectado tem de ser reciclado, mas o sistema operativo do servidor não tem de ser reiniciado.
  
**O que é o ASP.NET?**    
O ASP.NET é uma colecção de tecnologias dentro do Microsoft .NET Framework que permite que os programadores construam aplicações Web e Serviços Web XML.
  
Ao contrário das páginas Web tradicionais, que usam uma combinação de HTML estático e execução de scripts, o ASP.NET utiliza páginas compiladas e conduzidas por eventos. Como o ASP.NET é um ambiente de aplicação baseado na Web, requer um servidor Web para disponibilizar funcionalidades HTTP básicas, sendo executado sobre o Internet Information Services (IIS). Para obter mais informações, consulte o [site oficial Microsoft ASP.NET](http://www.asp.net/).
  
**Qual é a diferença entre Modo Integrado e Modo Clássico no IIS 7.0?**    
No IIS 7.0, cada Web site alojado no servidor pode ser executado no Modo Integrado ou no Modo Clássico. O Modo Integrado é o modo predefinido no IIS 7.0 e fornece controlo mais gradual sobre o processo de pedido. O Modo Clássico reverte o comportamento para o IIS 6.0 quando o Modo Integrado não está disponível. Para mais informações, consulte [How to Take Advantage of the IIS 7.0 Integrated Pipeline](http://learn.iis.net/page.aspx/244/how-to-take-advantage-of-the-iis7-integrated-pipeline).
  
**O que é um conjunto aplicacional?**    
No IIS 7.0, um conjunto aplicacional é um grupo de um ou mais URLs servidos por um processo de trabalho ou por um conjunto de processos de trabalho. Os conjuntos aplicacionais definem limites para as aplicações que contêm, o que significa que qualquer aplicação executada fora de um determinado conjunto aplicacional não afecta as aplicações executadas dentro desse mesmo conjunto aplicacional. Para mais informações, consulte [IIS 7.0: Managing Application Pools in IIS 7.0](http://technet.microsoft.com/en-us/library/cc753449(ws.10).aspx).
  
**Como poderia um intruso explorar a vulnerabilidade?**    
Um intruso poderia explorar esta vulnerabilidade criando uma série de pedidos HTTP anónimos especialmente concebida para o efeito para um sistema afectado e assim fazer com que o servidor Web a ser executado nesse sistema deixasse de responder.
  
**Quais os sistemas mais susceptíveis face a esta vulnerabilidade?**    
Os servidores nos quais o ASP 2.0 está hospedado no IIS 7.0 no modo integrado estão em risco.
  
**O que faz a actualização?**    
A actualização modifica a forma como o ASP.NET gere o agendamento de pedidos.
  
**Quando este boletim de segurança foi publicado, esta vulnerabilidade já tinha sido divulgada publicamente?**    
Não. A Microsoft recebeu informações sobre esta vulnerabilidade através de divulgação responsável. A Microsoft ainda não tinha recebido informações que indicassem que esta vulnerabilidade tivesse sido divulgada publicamente, quando este boletim de segurança foi publicado pela primeira vez.
  
**Quando este boletim de segurança foi publicado, a Microsoft já tinha recebido relatórios indicando que esta vulnerabilidade estava a ser explorada?**    
Sim. A Microsoft tem conhecimento de ataques limitados e muito específicos que tentam explorar a vulnerabilidade comunicada.

#### Informações de Implementação
  
**Instalação da actualização**
  
Quando instala esta actualização de segurança, o programa de instalação verifica se um ou mais ficheiros que estão a ser actualizados no seu sistema foram alvo de uma actualização anterior através de um hotfix da Microsoft.
  
Para mais informações sobre a terminologia que é usada neste boletim, tal como *hotfix*, consulte o [Artigo 824684 da Base de Dados de Conhecimento da Microsoft](http://support.microsoft.com/kb/824684/pt).
  
Esta actualização de segurança suporta os seguintes parâmetros de configuração.
  
| Parâmetro         | Descrição                                                                                                                                            |  
|-------------------|------------------------------------------------------------------------------------------------------------------------------------------------------|  
| **/?, /h, /help** | Apresenta ajuda para os parâmetros suportados.                                                                                                       |  
| **/quiet**        | Elimina a janela de mensagem de estado ou de erro.                                                                                                   |  
| **/norestart**    | Quando combinado com **/quiet**, o sistema não será reiniciado depois da instalação, mesmo que seja necessário reiniciar para concluir a instalação. |
  
**Nota** Para mais informações sobre o programa de instalação wusa.exe, consulte o [Artigo 934307 da Base de Dados de Conhecimento da Microsoft](http://support.microsoft.com/kb/934307).
  
**Verificar se a actualização foi aplicada**
  
-   **Microsoft Baseline Security Analyzer**
  
    Para verificar se uma actualização de segurança foi aplicada num sistema afectado, poderá utilizar a ferramenta Microsoft Baseline Security Analyzer (MBSA). Consulte a secção "**Orientações e ferramentas de detecção e implementação**" mais acima neste boletim para obter mais informações.
  
-   **Verificação das Versões dos Ficheiros**
  
    Dado existirem várias edições do Microsoft Windows, os passos que se seguem podem ser diferentes no seu sistema. Caso efectivamente o sejam, consulte a documentação do produto para executar estes passos.
  
    1.  Clique em **Iniciar** e depois introduza um nome de ficheiro de actualização em **Iniciar pesquisa**.  
    2.  Quando o ficheiro aparecer em **Programas**, clique com o botão direito do rato no nome do ficheiro e depois em **Propriedades**.  
    3.  No separador **Geral**, compare o tamanho do ficheiro com as tabelas de informação de ficheiros fornecidas no artigo da Base de Dados de Conhecimento do boletim.  
    4.  Também pode clicar no separador **Detalhes** e comparar informações, como a versão de ficheiro e a data de modificação, com as tabelas de informação fornecidas no artigo da Base de Dados de Conhecimento do boletim.  
    5.  Por fim, também pode clicar no separador **Versões anteriores** e comparar a informação de ficheiros sobre a versão anterior do ficheiro com a informação da versão do ficheiro nova ou actualizada.
