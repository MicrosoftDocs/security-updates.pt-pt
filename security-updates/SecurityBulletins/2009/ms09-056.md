Security Bulletin

Microsoft Security Bulletin MS09-056 - Importante
=================================================

Vulnerabilidades no Windows CryptoAPI poderiam permitir Spoofing (974571)
-------------------------------------------------------------------------

Data de publicação: 13 de outubro de 2009 | Updated: 14 de outubro de 2009

**actualizada:** 1.0

### Informações Gerais

#### Resumo Executivo

Esta actualização de segurança resolve duas vulnerabilidades do Microsoft Windows divulgadas publicamente. As vulnerabilidades poderiam permitir falsificações se um intruso obtivesse acesso ao certificado usado pelo utilizador final para autenticação.

Esta actualização de segurança está classificada como Importante para todas as edições suportadas do Windows 2000, Windows XP, Windows Server 2003, Windows Vista, Windows Server 2008 e Windows 7. Para mais informações, consulte a subsecção **Software Afectado e Software Não Afectado**, nesta secção.

A actualização de segurança resolve as vulnerabilidades modificando o CryptoAPI para rejeitar nomes de certificados que contêm terminadores inválidos, e para validar correctamente identificadores de objectos ASN. 1. Para mais informações sobre as vulnerabilidades, consulte a subsecção de Perguntas Mais Frequentes quanto à vulnerabilidade específica na secção seguinte, **Informações sobre as vulnerabilidades**.

**Recomendação.** A maioria dos clientes tem a actualização automática activada e não necessitará de efectuar qualquer acção, porque esta actualização de segurança será transferida e instalada automaticamente. Os clientes que não tenham a actualização automática activada necessitam de verificar as actualizações e instalar esta actualização manualmente. Para informações sobre opções específicas de configuração da actualização automática, consulte o [Artigo 294871 da Base de Dados de Conhecimento da Microsoft](http://support.microsoft.com/kb/294871).

Para administradores e instalações empresariais, ou utilizadores finais que pretendam instalar esta actualização de segurança manualmente, a Microsoft recomenda que os clientes apliquem a actualização na primeira oportunidade, utilizando software de gestão de actualizações ou verificando as actualizações através do serviço [Microsoft Update](http://update.microsoft.com/microsoftupdate/v6/default.aspx?ln=pt-pt).

Consulte também a secção **Orientações e Ferramentas de Detecção e Implementação**, mais à frente neste boletim.

**Questões conhecidas.** O [Artigo 974571 da Base de Dados de Conhecimento da Microsoft](http://support.microsoft.com/kb/974571) documenta as questões actualmente conhecidas que os clientes podem verificar quando instalam esta actualização de segurança. O artigo também documenta soluções recomendadas para estas questões. Quando as questões actualmente conhecidas e soluções recomendadas software dizem respeito apenas às edições específicas deste software, este artigo não fornece ligações a outros artigos.

Vulnerabilidade de Truncagem Inválida no Nome Comum de X. 509 - CVE-2009-2510  
-----------------------------------------------------------------------------
  
<span></span>
Existe uma vulnerabilidade de spoofing no componente CryptoAPI do Microsoft Windows durante a análise de informação ASN. 1 de certificados X. 509. Um intruso que explorasse esta vulnerabilidade com sucesso poderia fazer-se passar por outro utilizador ou sistema.
  
Para visualizar esta vulnerabilidade como uma entrada comum na lista de Vulnerabilidades e Exposições Comuns, consulte [CVE-2009-2510](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-2510).

#### Perguntas Mais Frequentes sobre a Vulnerabilidade de Truncagem inválida no Nome Comum de X. 509 - CVE-2009-2510
  
**Qual é a abrangência desta vulnerabilidade?**    
Esta é uma vulnerabilidade de ocultação (spoofing). Quando combinado com outros ataques específicos, tais como falsificação (spoofing) do DNS, isto pode permitir que um intruso falsifique com sucesso o certificado digital de um Web site para qualquer aplicação que utilize o CryptoAPI.
  
**O que provoca a vulnerabilidade?**    
O Windows CryptoAPI analisa incorrectamente um terminador inválido tendo como o final de qualquer um dos valores identificados por um Identificador de Objectos (OID). Isto faz com que o CryptoAPI analise apenas parte do valor real. Quando este valor parcial tem significado para o utilizador final, isto pode provocar uma situação em que um intruso poderia personificar outro utilizador ou sistema.
  
**O que é o Windows CryptoAPI?**    
O CryptoAPI fornece serviços que permite aos programadores de aplicações adicionar encriptação/desencriptação de dados, autenticação utilizando certificados digitais, e codificando e descodificando as suas aplicações baseadas em Windows a partir de Abstract Syntax Notation One (ASN.1). Os programadores de aplicações podem utilizar as funções no CryptoAPI sem conhecerem os detalhes da implementação subjacente, da mesma forma como podem utilizar uma biblioteca de gráficos desconhecendo a configuração de hardware dos gráficos em particular. O CryptoAPI funciona com um número de Cryptographic Service Providers que executa as funções criptográficas reais.
  
**O que é o ASN.1?**    
O Abstract Syntax Notation 1 (ASN.1) é uma norma de dados utilizada por muitas aplicações e dispositivos na indústria da tecnologia para permitir a normalização e compreensão de dados através de várias plataformas. O ASN. 1 não tem relação directa com qualquer norma, método de codificação, linguagem de programação ou plataforma de hardware específicos. É simplesmente uma linguagem para definir normas; as normas são escritas em ASN. 1.
  
Para mais informações sobre o ASN.1, consulte o [Artigo 252648 da Base de Conhecimentos da Microsoft](http://support.microsoft.com/kb/252648).
  
**O que é a norma X.509?**    
A especificação X. 509 define uma norma para gerir chaves públicas através de uma infraestrutura de chave pública (PKI). As chaves públicas são mantidas em certificados X. 509, que são documentos digitais que associam as informações de identidade de um utilizador a uma chave pública a partir de um par de chaves assimétricas públicas/privadas. As informações de identidade são normalmente compreensíveis pelas pessoas, como o nome completo ou endereço de correio electrónico de uma pessoa, ou o nome de anfitrião ou de domínio de uma máquina. Os certificados X. 509 são autorizados e publicados por um parceiro fidedigno, conhecido como uma Autoridade de Certificação (CA).
  
**O que é um Identificador de Objectos ou OID?**    
Os Identificadores de Objectos (OIDs) são valores numéricos publicados por várias autoridades para identificar unicamente elementos de dados, sintaxes e outras partes de aplicações distribuídas. Os OIDs podem ser encontrados em aplicações de OSI, directórios X. 500, SNMP e outras aplicações onde a imparidade é importante. Os OIDs são baseados numa estrutura em árvore, na qual uma autoridade superior, tal como ISO, atribui um ramo da árvore a uma sub-autoridade, que por sua vez pode criar sub-ramos.
  
**O que é um terminador INVÁLIDO?**    
Um terminador inválido é um caracter com o valor zero. Em determinadas linguagens de programação, tal como C, é utilizado como um caracter reservado para significar o fim de uma cadeia.
  
**Como poderia um intruso utilizar a vulnerabilidade?**    
Um intruso pode ser capaz de personificar outro utilizador ou sistema. As aplicações que utilizam o CryptoAPI para analisar estes dados podem devolver informações ao utilizador que poderiam fazer com que o utilizador tomasse decisões incorrectas.
  
**Como poderia um intruso explorar a vulnerabilidade?**    
Um intruso poderia tirar partido desta vulnerabilidade convencendo uma Autoridade de Certificação na qual o cliente confia a assinar um certificado contendo um terminador inválido no campo de Nome Comum (CN). O intruso poderia então configurar um Web site não autorizado que fornecesse este certificado a um computador Windows. O computador analisaria então incorrectamente o campo Nome Comum e apresentaria informações incorrectas ao utilizador.
  
**Quais os sistemas mais susceptíveis a esta vulnerabilidade?**    
Os clientes que utilizam Web sites HTTPs, ou que utilizam X. 509 para certificados, por exemplo, para proteger conversações de correio electrónico, estão mais susceptíveis a esta vulnerabilidade.
  
**O que faz a actualização?**    
Esta actualização de segurança resolve a vulnerabilidade modificando o CryptoAPI para rejeitar Nomes de Certificados que contêm terminadores inválidos.

#### Perguntas Mais Frequentes sobre a Vulnerabilidade de Excesso de Número Inteiro em Identificadores de Objectos X.509 - CVE-2009-2511
  
**Qual é a abrangência desta vulnerabilidade?**    
Esta é uma vulnerabilidade de ocultação (spoofing). Quando combinado com outros ataques específicos, tais como falsificação (spoofing) do DNS, isto pode permitir que um intruso falsifique com sucesso o certificado digital de um Web site para qualquer aplicação que utilize o CryptoAPI.
  
**O que provoca a vulnerabilidade?**    
O Windows CryptoAPI tem uma sobrecarga de número inteiro quando analisa identificadores de objecto ASN. 1 de certificados X. 509, tais como Nome Comum (CN). Um intruso poderia gerar um certificado malicioso que seria analisado incorrectamente pelo Windows CryptoAPI, permitindo que o intruso personificasse outro utilizador ou sistema.
  
**O que é o Windows CryptoAPI?**    
O CryptoAPI fornece serviços que permite aos programadores de aplicações adicionar encriptação/desencriptação de dados, autenticação utilizando certificados digitais, e codificando e descodificando as suas aplicações baseadas em Windows a partir de Abstract Syntax Notation One (ASN.1). Os programadores de aplicações podem utilizar as funções no CryptoAPI sem conhecerem os detalhes da implementação subjacente, da mesma forma como podem utilizar uma biblioteca de gráficos desconhecendo a configuração de hardware dos gráficos em particular. O CryptoAPI funciona com um número de Cryptographic Service Providers que executa as funções criptográficas reais.
  
**O que é o ASN.1?**    
O Abstract Syntax Notation 1 (ASN.1) é uma norma de dados utilizada por muitas aplicações e dispositivos na indústria da tecnologia para permitir a normalização e compreensão de dados através de várias plataformas. O ASN. 1 não tem relação directa com qualquer norma, método de codificação, linguagem de programação ou plataforma de hardware específicos. É simplesmente uma linguagem para definir normas; as normas são escritas em ASN. 1.
  
Para mais informações sobre o ASN.1, consulte o [Artigo 252648 da Base de Conhecimentos da Microsoft](http://support.microsoft.com/kb/252648).
  
**O que é a norma X.509?**    
A especificação X. 509 define uma norma para gerir chaves públicas através de uma infraestrutura de chave pública (PKI). As chaves públicas são mantidas em certificados X. 509, que são documentos digitais que associam as informações de identidade de um utilizador a uma chave pública a partir de um par de chaves assimétricas públicas/privadas. As informações de identidade são normalmente compreensíveis pelas pessoas, como o nome completo ou endereço de correio electrónico de uma pessoa, ou o nome de anfitrião ou de domínio de uma máquina. Os certificados X. 509 são autorizados e publicados por um parceiro fidedigno, conhecido como uma Autoridade de Certificação (CA).
  
**O que é um Identificador de Objectos ou OID?**    
Os Identificadores de Objectos (OIDs) são valores numéricos únicos publicados por várias autoridades para identificar unicamente elementos de dados, sintaxes e outras partes de aplicações distribuídas. Os OIDs podem ser encontrados em aplicações de OSI, directórios X. 500, SNMP e outras aplicações onde a imparidade é importante. Os OIDs são baseados numa estrutura em árvore, na qual uma autoridade superior, tal como ISO, atribui um ramo da árvore a uma sub-autoridade, que por sua vez pode criar sub-ramos.
  
**Como poderia um intruso utilizar a vulnerabilidade?**    
Um intruso pode ser capaz de personificar outro utilizador ou sistema. As aplicações que utilizam o CryptoAPI para analisar estes dados podem devolver informações ao utilizador que poderiam fazer com que o utilizador tomasse decisões incorrectas.
  
**Como poderia um intruso explorar a vulnerabilidade?**    
Um intruso poderia explorar esta vulnerabilidade convencendo uma Autoridade de Certificação na qual o utilizador confia a assinar um certificado contendo um identificador de objecto mal formado. O intruso poderia então configurar um Web site que fornecesse este certificado a um cliente Windows. Esta aplicação de cliente analisaria então incorrectamente o campo Nome Comum e ofereceria informações de fidedignidade incorrectas ao utilizador.
  
**Quais os sistemas mais susceptíveis a esta vulnerabilidade?**    
As aplicações de clientes que utilizam Web sites HTTPs, ou que utilizam X. 509 para certificados, por exemplo, para proteger conversações de correio electrónico, estão mais susceptíveis a esta vulnerabilidade.
  
**O que faz a actualização?**    
Esta actualização de segurança resolve a vulnerabilidade fazendo com que o CryptoAPI valide correctamente os identificadores de objectos ASN. 1.
  
**Quando este boletim de segurança foi publicado, esta vulnerabilidade já tinha sido divulgada publicamente?**    
Sim. Esta vulnerabilidade foi divulgada publicamente. Foi-lhe atribuído o número de Vulnerabilidade e Exposição Comum [CVE-2009-2511](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-2511).
  
**Quando este boletim de segurança foi publicado, a Microsoft já tinha recebido informações sobre a exploração desta vulnerabilidade?**    
Não. A Microsoft não tinha recebido quaisquer informações que indicassem que esta vulnerabilidade tivesse sido utilizada publicamente para atacar clientes e não tinha conhecimento de exemplos publicados de código para prova de conceito quando este boletim de segurança foi criado pela primeira vez.

#### Informações de Implementação

**Instalação da actualização**

Quando instala esta actualização de segurança, o programa de instalação verifica se um ou mais ficheiros que estão a ser actualizados no seu sistema foram alvo de uma actualização anterior através de um hotfix da Microsoft.

Caso tenha instalado anteriormente um hotfix para actualizar um destes ficheiros, o programa de instalação copia os ficheiros RTMQFE, SP1QFE ou SP2QFE para o sistema.     Dado existirem várias edições do Microsoft Windows, os passos que se seguem podem ser diferentes no seu sistema. Caso efectivamente o sejam, consulte a documentação do produto para executar estes passos.

    1.  Clique em **Iniciar** e, em seguida, em **Procurar**.
    2.  No painel **Resultados da Procura**, clique em **Todos os ficheiros e pastas** no **Assistente de Pesquisa**.
    3.  Na caixa **Totalidade ou parte do nome de ficheiro**, escreva um nome de ficheiro a partir da tabela de informações de ficheiros adequada e, em seguida, clique em **Procurar**.
    4.  Na lista de ficheiros, clique com o botão direito do rato no nome de ficheiro a partir da tabela de informações de ficheiros adequada e, em seguida, clique em **Propriedades**.

    5.  No separador **Versão**, determine a versão do ficheiro que está instalada no sistema, comparando-a com a versão que é documentada na tabela de informações de ficheiros adequada.
